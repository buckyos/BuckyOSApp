# BuckyOS App 托盘图标需求



## 1. 背景与定位

托盘图标是 **BuckyOS Desktop 版本特有的桌面能力**。它的核心作用是：让用户在当前桌面设备上快速查看 BuckyOS 的本机运行状态，并进入少量本机级快捷操作。

托盘图标跟随 **BuckyOS App** 的生命周期与配置：

- 当 BuckyOS App 已启动，且开启 `Enable Tray Icon / 启用托盘图标` 配置时，托盘图标出现。
- 当 BuckyOS App 退出，或用户关闭 `Enable Tray Icon / 启用托盘图标` 配置时，托盘图标不出现。
- 托盘图标本身属于 BuckyOS App 的控制入口，不应被理解为整个 BuckyOS 分布式集群的总开关。

## 2. 核心边界：只能管理当前设备

BuckyOS 是分布式集群系统，因此必须明确：**托盘图标只能表达和管理当前这台设备上的 BuckyOS 状态**。

用户在某台笔记本或桌面电脑的系统托盘区看到的图标，原则上只能控制这台设备上的服务。即使菜单中出现“停止”“启动”等操作，也必须解释为：

> 停止或启动当前设备上的 BuckyOS 相关本地服务，而不是停止或启动整个 BuckyOS 集群。

这是一条非常重要的产品边界。不能让用户误以为：在某台电脑的托盘图标里点击“停止”，就会把整个 BuckyOS 集群停掉。

## 3. 部署形态与控制对象

托盘图标的具体含义会随当前设备安装的 BuckyOS 形态不同而变化，但边界始终是“当前设备”。

### 3.1 单机 Desktop Service 形态

在当前版本或早期形态中，BuckyOS Desktop Service 可以作为长期运行的单节点 BuckyOS 存在。

例如用户在一台笔记本上安装了完整的 BuckyOS Desktop，所有服务都运行在这台设备上。在这种情况下：

- 托盘图标显示的是这台设备上 Desktop Service 的状态。
- 菜单中的“启动 / 停止”本质上是启动或停止这台设备上的 Desktop Service。
- 因为所有服务都在同一台设备上，所以停止本机服务在用户感知上可能等价于停止整个本地单机 BuckyOS。

这属于特殊情况。即便如此，文案仍应强调“本机”或“当前设备”，避免和未来集群形态混淆。

### 3.2 标准集群客户端 / 设备控制器形态

未来更标准的形态是：用户拥有一套 BuckyOS 集群或搭载 BuckyOS 的硬件，并把笔记本、Windows 工作站等桌面设备作为成员设备加入集群。

在这种情况下，桌面设备上可能运行：

- Gateway 类服务，用于保证本机与集群之间的通道可用；
- Node Daemon / Device Daemon 类服务，用于把当前设备能力提供给集群；
- 其他本机级客户端服务。

此时托盘图标表达的是：**当前设备上这些本地 BuckyOS 服务的状态**。它不是集群控制台，也不是整个集群的启停入口。

## 4. 图标状态

托盘图标需要让用户一眼看出当前设备上的 BuckyOS 状态。基础状态至少包括以下三类。

| 状态 | 含义 | 图标表达建议 |
|---|---|---|
| 正常运行 | 当前设备上的 BuckyOS 相关服务正在正常运行 | 正常态图标，可使用正常状态色 |
| 已停止 | 当前设备上的 BuckyOS 相关服务处于停止状态 | 停止态图标，可使用灰色或弱化样式 |
| 异常 / 有问题 | 系统按配置本应运行，但服务没有正常启动、进程异常或健康检查未通过 | 图标叠加感叹号或警告标记 |

建议补充过程状态，用于启动或停止操作之后的短暂阶段：

- 启动中；
- 停止中；
- 状态刷新中。

这些过程状态可以在菜单的详细说明中体现，托盘主图标可根据设计选择是否单独表现。

## 5. 图标形态：区分运行状态与部署形态

图标应同时传达两类信息：

1. 当前设备上的服务状态：运行、停止、异常；
2. 当前设备的 BuckyOS 形态：单机 Desktop Service，或标准集群中的本机客户端 / 设备节点。

状态颜色应优先用于表达“运行 / 停止 / 异常”。部署形态建议通过图标背景、底板、角标或徽标区分。

设计倾向：

- 用图标主体或颜色表达运行状态；
- 用背景或角标表达部署形态；
- 让用户能快速识别：当前点击“停止”影响的是“本机服务”，还是在单机形态下会影响整套本地 Desktop Service。

具体视觉规则可由设计定稿，但产品上需要保留这个区分能力。

## 6. 点击与菜单交互

### 6.1 点击行为

托盘图标不区分左键点击和右键点击。

原因是很多笔记本和触摸板环境下，右键操作并不方便。因此：

- 鼠标左键点击托盘图标，应打开同一个菜单；
- 鼠标右键点击托盘图标，也应打开同一个菜单；
- 触摸板点击或系统等价点击，也应打开同一个菜单。

### 6.2 菜单结构

菜单需要保持简单，不应做成复杂控制台。建议结构如下：

```text
[当前状态详细说明，非操作项]
--------------------------------
操作 >
  启动本机 BuckyOS 服务
  停止本机 BuckyOS 服务
  重启本机 BuckyOS 服务（可选）
  ...
打开 BuckyOS App
BuckyOS 设置（可选）
关于 BuckyOS / 查看信息（可选）
--------------------------------
退出 BuckyOS App
```

说明：

- 最顶部必须是当前状态的详细解释，且是不可操作项。
- 第一根分隔线之后是操作区。
- 启动、停止等服务操作放在二级菜单中，避免用户误触。
- 操作二级菜单中可以列出所有操作，但不适用于当前状态的操作应置灰禁用。
- 再往下是快捷入口，例如打开 BuckyOS App、进入设置、查看关于信息等。
- 最底部是退出入口。
- 菜单整体建议只保留两根分隔线：状态说明之后一根，退出之前一根。

## 7. 状态说明文案

菜单顶部的状态说明需要明确、可理解，并始终强调“当前设备”。

示例：

### 正常运行

> BuckyOS 正在当前设备上正常运行。

标准集群形态可补充：

> 当前设备上的 BuckyOS Gateway / Device 服务运行正常。此状态不代表整个集群所有设备的状态。

### 已停止

> 当前设备上的 BuckyOS 本地服务已停止。

单机 Desktop Service 形态可补充：

> 此设备运行的是单机 Desktop Service。停止后，本机上的 BuckyOS 服务不可用。

### 异常 / 有问题

> 当前设备上的 BuckyOS 服务未能正常运行。系统配置显示它应处于运行状态，但健康检查未通过或相关进程未启动。

## 8. 操作区需求

### 8.1 操作放入二级菜单

启动、停止等操作对用户影响较大，应故意放深一层。用户点击托盘图标后，不应直接看到一键停止按钮，而是通过“操作 >”进入二级菜单。

### 8.2 操作项展示规则

操作二级菜单中可以始终展示完整操作集合，例如：

- 启动本机 BuckyOS 服务；
- 停止本机 BuckyOS 服务；
- 重启本机 BuckyOS 服务（如需要）；
- 重新检查状态（如需要）。

根据当前状态决定哪些操作可点击，哪些操作置灰。

示例：

| 当前状态 | 可点击操作 | 置灰操作 |
|---|---|---|
| 正常运行 | 停止本机服务、重启本机服务 | 启动本机服务 |
| 已停止 | 启动本机服务 | 停止本机服务、重启本机服务 |
| 异常 | 重试启动、停止本机服务、重新检查状态 | 视具体状态决定 |

### 8.3 二次确认

用户点击启动、停止等操作后，必须出现二次确认。确认文案应根据当前部署形态解释操作后果。

#### 标准集群客户端 / 设备控制器形态

停止确认示例：

> 此操作只会停止当前设备上的 BuckyOS 本地服务，不会停止整个 BuckyOS 集群。是否继续？

启动确认示例：

> 此操作会启动当前设备上的 BuckyOS 本地服务，不会影响其他设备。是否继续？

#### 单机 Desktop Service 形态

停止确认示例：

> 此设备运行的是单机 BuckyOS Desktop Service。停止后，本机上的 BuckyOS 服务将不可用。是否继续？

启动确认示例：

> 此操作会启动本机 BuckyOS Desktop Service。是否继续？

## 9. 快捷入口

### 9.1 打开 BuckyOS App

托盘菜单应提供“打开 BuckyOS App”入口。

背景：BuckyOS App 的进程可能仍在运行，但主窗口或 UI 已关闭 / 隐藏。用户可以通过托盘图标重新拉起 App UI。

要求：

- 点击后打开或恢复 BuckyOS App 主界面；
- 如果主界面已打开，则将其置前；
- 如果 App UI 已隐藏，则重新显示。

### 9.2 设置、关于与信息入口

可在快捷入口区加入：

- BuckyOS 设置；
- 查看当前设备信息；
- 关于 BuckyOS；
- 其他未来常用入口。

这些入口不是第一版本的核心重点，可按实现节奏逐步加入。

### 9.3 常见 App 入口

过去设计中曾考虑让托盘菜单成为常见 App 的入口。当前版本不需要复杂化，只需预留未来扩展可能。

第一版本建议先保持简单：状态说明、操作二级菜单、打开 BuckyOS App、退出。

## 10. 退出入口

菜单底部必须提供“退出 BuckyOS App”入口。

退出的含义必须明确：

> 退出 BuckyOS App / 托盘图标控制程序，不代表停止 BuckyOS 本地服务，更不代表停止整个 BuckyOS 集群。

建议在用户首次点击退出时展示确认文案，或在退出项旁用更准确的命名避免误解，例如：

- 退出 BuckyOS App；
- 退出托盘程序；
- 关闭 BuckyOS App，不停止服务。

## 11. 状态聚合规则建议

托盘图标需要从本机服务状态聚合出一个清晰状态。

建议规则：

1. 当前设备所有必要服务均正常运行：显示“正常运行”。
2. 当前设备服务被用户主动停止，且没有配置要求其自动运行：显示“已停止”。
3. 系统配置要求服务运行，但必要服务未启动、健康检查失败或进程异常退出：显示“异常 / 有问题”。
4. 如果当前设备上存在可选服务，可选服务未运行不应直接触发异常，除非配置中明确要求该服务运行。
5. 标准集群形态下，托盘图标不应因为其他设备异常而直接变成异常；除非该异常影响当前设备的本地连接或本地服务。

## 12. 第一版本范围建议

第一版本优先实现以下能力：

- BuckyOS App 启动且启用托盘配置时显示托盘图标；
- App 退出或关闭托盘配置时隐藏托盘图标；
- 支持运行、停止、异常三种基础图标状态；
- 支持通过背景、角标或其他视觉方式区分单机 Desktop Service 与标准集群客户端形态；
- 任意点击托盘图标打开同一菜单；
- 菜单顶部显示当前状态详细说明；
- 服务操作放入二级菜单；
- 启动 / 停止操作有二次确认，并明确影响范围仅限当前设备；
- 提供“打开 BuckyOS App”入口；
- 提供“退出 BuckyOS App”入口，且退出不停止服务。

## 13. 非目标

以下内容不属于托盘图标的直接目标：

- 通过托盘图标启动、停止或重启整个 BuckyOS 集群；
- 通过托盘图标替代完整的集群管理控制台；
- 在第一版本中提供复杂 App 启动器；
- 在第一版本中展示完整集群拓扑、所有节点健康详情或远程节点控制能力。

## 14. 验收标准

1. 当 BuckyOS App 启动且启用托盘图标时，系统托盘中能看到图标。
2. 当 BuckyOS App 退出或关闭启用项时，托盘图标消失。
3. 托盘图标能正确反映当前设备上的 BuckyOS 本地服务状态。
4. 异常状态应使用明显的感叹号或警告标记。
5. 点击托盘图标，无论左键、右键或触摸板点击，都打开同一菜单。
6. 菜单第一项是当前状态详细说明，且不可点击。
7. 启动、停止等操作位于二级菜单，不直接暴露在一级菜单中。
8. 不适用于当前状态的操作应置灰禁用。
9. 点击启动或停止后必须有二次确认。
10. 二次确认文案必须说明操作只影响当前设备；单机 Desktop Service 形态下需说明其特殊影响。
11. “打开 BuckyOS App”可以恢复或置前 App 主界面。
12. “退出 BuckyOS App”只退出 App / 托盘控制程序，不停止本机服务。

## 15. 待确认事项

- 具体视觉方案：哪种背景、角标或徽标用于区分部署形态。
- 基础状态颜色规范：运行、停止、异常分别使用哪些颜色。
- 本机服务的准确进程列表与健康检查规则。
- 是否在第一版本中支持“重启本机服务”“重新检查状态”等操作。
- 设置、关于、设备信息入口是否进入第一版本。
- 退出 BuckyOS App 是否需要每次二次确认，还是仅首次提示。

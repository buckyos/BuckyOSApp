# BuckyOS App 需求文档大纲（中文）

## 1. 文档目的

本文档用于整理 BuckyOS App 的需求边界、核心能力、用户流程与接口约束，作为后续补充正式需求说明、产品设计、开发排期与对外协作的基础版本。

当前整理基于现有项目代码与已有零散文档，重点围绕以下两条主线展开：

- 身份管理：负责 DID 的创建、导入、切换、备份、删除，以及与 SN/OOD 相关的身份绑定。
- Runtime 提供：作为 OOD 内 App 的宿主运行环境，为 Web App / iframe / 外部页面提供统一运行容器与受控能力接口。

## 2. 项目背景与定位

### 2.1 项目背景

- BuckyOS 生态中的 App 需要一个统一入口管理本地身份。
- OOD 内部 App 或通过 Runtime 承载的页面，需要安全访问宿主能力，而不是直接接触底层实现。
- 当前项目采用 Tauri + React + Rust 实现，兼顾桌面端与移动端扩展能力。

### 2.2 项目定位

BuckyOS App 是一个面向终端用户的宿主应用，承担以下职责：

- 管理用户在本机的 DID 身份与密钥派生结果。
- 帮助用户完成 SN 账号注册/绑定。
- 帮助用户发现并激活 OOD 设备。
- 作为 App Runtime 宿主，为 OOD 内 App 或外部页面提供受控的能力调用入口。
- 提供应用列表展示与后续应用入口管理能力。

### 2.3 目标与非目标

#### 目标

- 为用户提供完整、可用、可恢复的 DID 身份管理能力。
- 为 OOD 内 App 提供统一 Web Runtime 容器。
- 为第三方页面提供标准化的 `BuckyApi` 能力桥接。
- 形成 SN 绑定、OOD 激活、App 访问之间的闭环。

#### 非目标

- 不在第一阶段承担完整的应用商店能力。
- 不在第一阶段实现复杂的权限中心或细粒度授权体系。
- 不在第一阶段覆盖所有原生能力，只先开放少量核心接口。

## 3. 术语说明

- DID：用户的去中心化身份。
- Bucky Wallet：由助记词派生出的 Bucky 身份钱包，用于 DID 标识与签名。
- SN：与用户名注册、身份查询相关的服务节点能力。
- OOD：用户侧设备或节点，用于承载应用与相关运行环境。
- Runtime：宿主提供的应用运行容器，支持独立 WebView 或嵌入式 iframe 方式运行页面。
- BuckyApi：宿主注入给页面的 JS API，用于页面调用受控宿主能力。

## 4. 用户角色与典型场景

### 4.1 终端用户

- 首次安装并创建或导入身份。
- 绑定 SN 用户名。
- 激活或连接 OOD 设备。
- 在 Runtime 中打开和使用 App。
- 管理已有身份，执行切换、备份、删除等操作。

### 4.2 第三方 App 开发者

- 将应用运行在 BuckyOS Runtime 中。
- 通过 `window.BuckyApi` 获取当前用户身份信息。
- 发起 JSON 签名请求以完成认证或业务签名。

### 4.3 平台/系统集成人员

- 配置应用列表来源。
- 定义 Runtime 对外能力边界。
- 规划与 BuckyOS 目录结构、包管理、设备环境的衔接方式。

## 5. 产品范围

### 5.1 已有或明确在做的能力范围

- DID 创建、导入、激活、切换、删除。
- 助记词展示与备份确认。
- 多钱包扩展派生能力。
- SN 用户名校验、邀请码校验、SN 绑定与状态缓存。
- OOD 扫描发现与激活入口。
- 应用列表读取与展示。
- Web Runtime 容器与新窗口打开能力。
- 宿主向页面注入 `BuckyApi` 并处理消息桥接。

### 5.2 后续可扩展范围

- App 安装、升级、卸载与依赖管理。
- 更完整的宿主能力开放列表。
- 面向 OOD 内 App 的权限授权与审计机制。
- 与 BuckyOS `local/bin`、`bin`、`local/etc` 的更深度联动。

## 6. 核心需求模块

### 6.1 身份管理

#### 6.1.1 首次引导

- 用户首次进入时，若本地无身份，进入欢迎与引导流程。
- 支持“创建 DID”与“导入 DID”两种主入口。
- 需要明确展示 DID 的作用与后续使用场景。

#### 6.1.2 创建 DID

- 用户输入昵称、密码并生成助记词。
- 用户需完成助记词确认后，系统才可真正创建 DID。
- 创建成功后默认设为当前激活身份。
- 默认至少派生一枚 Bucky wallet。

#### 6.1.3 导入 DID

- 用户可通过助记词导入已有身份。
- 系统需校验助记词合法性。
- 系统需避免与已有 DID 重复导入。
- 导入成功后默认设为当前激活身份。

#### 6.1.4 身份列表与切换

- 系统需展示本地所有 DID。
- 用户切换身份时需要进行密码校验。
- 当前激活身份需要明确标识。
- 当系统存在身份但未激活时，应强制用户先选择一个身份。

#### 6.1.5 身份备份

- 用户可对当前身份执行助记词备份。
- 备份前必须校验密码。
- 备份流程应包含“展示助记词”和“确认助记词”两个步骤。

#### 6.1.6 身份删除

- 用户可删除当前身份。
- 删除前必须二次确认并校验密码。
- 删除当前激活身份后，系统需根据剩余身份数量决定回到欢迎页还是进入身份选择页。

#### 6.1.7 钱包扩展派生

- 在同一 DID 下，支持继续派生 Bucky、BTC、ETH 地址。
- 系统需保证派生索引连续、不可重复。
- 该能力当前更偏底层能力，是否在正式产品界面露出需再确认。

### 6.2 SN 绑定管理

#### 6.2.1 查询当前 DID 的 SN 状态

- 系统需根据当前激活 DID 的公钥查询 SN 绑定状态。
- 查询结果需支持本地缓存，减少重复请求。
- 查询失败时应允许用户重试。

#### 6.2.2 用户名与邀请码校验

- 用户名需满足长度、字符格式等规则。
- 用户名可用性需在服务端校验。
- 邀请码合法性需在服务端校验。

#### 6.2.3 SN 注册/绑定

- 用户填写用户名与邀请码后可发起绑定。
- 绑定成功后系统需轮询查询结果，直到 SN 记录可见或超时。
- 绑定结果需与 DID 关联并持久化保存。

### 6.3 OOD 激活与绑定

#### 6.3.1 激活入口

- 用户在完成 SN 绑定后，应能继续进入 OOD 激活流程。
- 若已检测到当前 DID 已绑定 OOD，应展示已绑定状态，而非继续引导激活。

#### 6.3.2 局域网扫描发现设备

- 系统需扫描本机所在网段，发现可响应的 OOD 设备。
- 设备列表需展示基础信息，如主机名、设备类型、访问地址等。
- 系统应支持识别当前设备自身 IP，对列表中的“本机设备”做标识。

#### 6.3.3 进入 OOD 激活页

- 用户点击扫描到的设备后，应打开对应的激活页面。
- 激活页需运行在宿主提供的 Runtime 容器中。
- 后续是否由本应用继续承接授权/写入流程，需要在正式需求中补充。

### 6.4 App 与 Runtime 管理

#### 6.4.1 App 列表

- 系统需能读取应用列表配置并展示可用应用。
- 需兼容开发模式与正式环境的不同列表来源。
- 应用列表至少包含显示名称、图标、提示信息、包名等元数据。

#### 6.4.2 App 打开方式

- 支持以独立 WebView 窗口打开 App。
- 支持以嵌入式 iframe 容器运行页面。
- 当同一 label 的窗口已存在时，应优先复用已有窗口，避免重复打开。

#### 6.4.3 Runtime 宿主职责

- 自动向页面注入 `window.BuckyApi`。
- 通过 `postMessage` 机制转发页面请求与宿主响应。
- 对页面开放统一 Promise 风格的调用协议。

### 6.5 宿主能力开放

#### 6.5.1 当前已开放接口

- `getPublicKey`
- `getCurrentUser`
- `signJsonWithActiveDid`

#### 6.5.2 接口要求

- 所有接口返回统一结构：`code`、`message`、`data`。
- 需要定义清晰、稳定的错误码。
- 需要保证无激活 DID、无公钥、密码错误、用户取消、并发签名冲突等场景可被页面识别。

#### 6.5.3 交互式签名

- 页面发起签名请求后，宿主需弹出密码输入框。
- 宿主完成密码校验后再调用本地 DID 能力签名。
- 宿主需避免并发签名流程重入。

### 6.6 配置、日志与环境兼容

- 系统需支持读取 SN API Host 等配置。
- 系统需具备日志记录能力，便于调试与问题追踪。
- 开发模式下应提供测试页面与调试入口。
- 架构上需兼容桌面端与移动端扩展。

## 7. 关键用户流程

### 7.1 首次使用流程

1. 打开应用。
2. 若本地无身份，进入欢迎页。
3. 用户选择创建 DID 或导入 DID。
4. 完成身份创建/导入后进入主界面。
5. 引导用户绑定 SN。
6. 引导用户激活 OOD。
7. 进入应用使用阶段。

### 7.2 已有身份用户流程

1. 打开应用。
2. 若本地已存在身份，直接进入主界面。
3. 用户可在设置中切换身份、备份身份、删除身份。
4. 用户可在首页查看 SN/OOD 状态。
5. 用户可在应用页查看可用 App。

### 7.3 第三方页面调用宿主能力流程

1. 页面运行于 Runtime。
2. 页面检测 `window.BuckyApi` 是否存在。
3. 页面发起身份查询或签名请求。
4. 宿主执行本地逻辑并返回标准结果。
5. 页面根据返回码完成业务处理。

## 8. 数据与状态管理要求

### 8.1 本地持久化数据

- DID 列表。
- 当前激活 DID。
- 加密保存的助记词/种子。
- SN 绑定状态缓存。
- 配置项与日志文件。

### 8.2 状态一致性要求

- 激活 DID 变化后，主界面、SN 状态、Runtime 暴露身份需同步更新。
- 删除 DID 后，应避免残留无效状态。
- 页面侧读取的当前用户信息需与宿主当前激活身份保持一致。

## 9. 非功能需求

### 9.1 安全

- 助记词必须加密存储。
- 敏感操作必须要求密码校验。
- 宿主能力不得无边界暴露给任意页面。
- 签名动作需具备明确的用户确认过程。

### 9.2 可用性

- 首次使用流程应尽量短且清晰。
- 网络异常、查询失败、绑定超时等场景需提供可理解的提示。
- 页面与原生交互需要有明确超时与错误反馈机制。

### 9.3 可维护性

- 接口协议与错误码需要稳定。
- 文档需覆盖页面、宿主桥接、Rust 命令之间的映射关系。
- App 列表、SN 服务地址等应尽量配置化。

## 10. 接口与集成要求

### 10.1 前后端边界

- React 层负责页面、状态管理、用户交互、iframe/窗口容器。
- Rust 层负责 DID、密钥派生、签名、配置、网络基础能力、应用列表读取。

### 10.2 对外文档

- 需要提供第三方开发者可直接使用的 `BuckyApi` 集成文档。
- 需要说明宿主环境判断方式、接口列表、错误码、示例代码。

### 10.3 与 BuckyOS 系统环境集成

- 应用列表读取优先来自 `BUCKYOS_ROOT/bin/applist.json`。
- 开发环境下允许回退到仓库内置配置。
- 后续需要补充与 `local/bin`、`local/etc`、app-daemon、native app 生命周期的关系说明。

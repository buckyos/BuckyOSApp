# BuckyOSApp 测试流程

本文档根据 [当前需求文档](../当前需求文档.md) 拆分当前版本验收流程。每个测试项都要求具备明确输入、自动化执行方式、分支覆盖和通过标准。

执行原则：

- 优先自动化。页面流程使用 Android 模拟器 + adb/UI 自动化或桌面端 Tauri 自动化；接口和持久化检查使用脚本或命令扫描。
- 能 mock 的外部依赖必须 mock。SN 正常、失败、超时、轮询可见、无记录等分支应通过本地 mock SN 服务覆盖，真实 SN 只作为联调冒烟。
- 不能完全自动化的项目必须在本文件中标注原因，并在测试报告中写明是“通过自动化”“部分自动化”还是“人工确认”。
- 所有截图、日志和报告不得写入完整助记词、明文密码、私钥 PEM、完整 `pwd_hash`、完整 Active Code。

## 0. 自动化环境约定

### 0.1 自动化能力分级

| 等级 | 含义 |
| --- | --- |
| 可完全自动化 | 可由脚本稳定完成输入、断言、截图和结果输出 |
| 可部分自动化 | 主流程可由脚本完成，但依赖真实设备、真实 SN 资源、桌面窗口焦点或人工观察 |
| 不可完全自动化 | 当前仓库缺少可控能力或平台限制导致无法稳定由脚本独立判定 |

### 0.2 推荐执行工具

- Android UI：Pixel 4 XL 模拟器，adb 启动、点击、输入、截图、UI tree dump、logcat 采集。
- 桌面端 UI：Tauri dev/build 启动后，用可控窗口脚本或 WebDriver/Playwright 类工具检查路由、截图和窗口数量。
- SN 分支：本地 mock SN JSON-RPC 服务，通过应用配置目录 `config.json` 的 `sn_host` 指向 mock base URL。
- 静态审查：`rg` 搜索敏感关键字、Rust/TS 单元测试、构建命令。
- 报告：按 [测试报告模板](./测试报告模板.md) 输出。

### 0.3 测试数据

- 统一密码：`111111`。
- 用户名：自动化生成，例如 `auto-${timestamp}`，实际提交前需要满足 `^[a-z][a-z0-9-]{5,}[a-z0-9]$`，因此推荐格式 `autotest${timestamp}`。
- Active Code：mock SN 环境使用固定值 `MOCK_ACTIVE_CODE_OK`；真实 SN 冒烟使用需求文档第 16 节列出的未使用邀请码。
- 无效 Active Code：`BAD_ACTIVE_CODE`。
- 无 SN 记录助记词：由脚本新生成但不注册到 mock SN 的助记词。
- 重复导入助记词：创建成功流程保存的同一组助记词，只允许在测试运行时内存保留，不写入报告。

### 0.4 执行前清理

自动化脚本每轮开始前应清理测试 App 数据，确保启动分流可重复：

- Android：清理应用数据或重新安装 debug 包。
- 桌面端：清理当前测试配置目录中的 `wallet.store`、测试用 `config.json` 和相关本地缓存。
- Web/localStorage：清理 `buckyos.locale`、`buckyos.theme`，除非当前用例专门验证持久化。

### 0.5 结果证据目录

建议每轮输出到独立目录：

```text
doc/test/runs/YYYYMMDD-HHMMSS/
```

目录内容：

- `report.md`：测试报告。
- `screenshots/`：页面截图。
- `logs/`：构建、adb、logcat、mock SN 请求日志。
- `artifacts/`：UI tree、持久化数据脱敏快照、静态扫描结果。

2. 启动 App。
3. 断言强制跳转身份切换页。

## 2. 创建账户流程

自动化等级：可完全自动化，前提是使用 mock SN；真实 SN 冒烟为可部分自动化。

### 2.1 创建账户主链路

步骤：

1. 清理 App 数据并启动。
2. 从欢迎页进入创建账户说明页。
3. 打开 DID 说明页后返回。
4. 点击 SN 说明卡片。
5. 点击主按钮生成助记词并进入助记词展示页。
6. 读取并临时保存 12 个助记词。
7. 点击确认进入助记词确认页。
8. 按正确顺序点击助记词。
9. 进入绑定 SN 页。
10. 输入合法用户名、密码、确认密码、有效 Active Code。
11. 等待 mock SN 返回用户名可用、Active Code 可用。
12. 提交。
13. mock SN 依次断言收到 `auth.register`、`user.bind_owner_key`、`device.get_by_pk` 轮询。
14. 断言进入创建成功页。
15. 点击主按钮进入 `/main/home/ood-activate`。

通过标准：

- 助记词为 12 个英文 BIP-39 单词。
- 助记词确认必须按顺序完成。
- SN 注册、owner key 绑定、SN 记录可见、本地 DID 创建全部成功后才进入成功页。
- 成功页展示昵称、BuckyOS DID，BTC/ETH 地址如果存在则展示。
- 创建成功后的 DID 自动成为 Active DID。
- 截图 `02-create-show-mnemonic.png`、`02-create-bind-sn.png`、`02-create-success.png`。

### 2.2 助记词确认错误顺序

步骤：

1. 进入助记词确认页。
2. 故意点击错误顺序。
3. 检查继续按钮状态。
4. 重置或退回后按正确顺序完成。

通过标准：

- 顺序错误时不能进入绑定 SN 页。
- 修正后可以继续。
- 截图 `02-mnemonic-wrong-order.png`。

### 2.3 创建流程中途退出不恢复

步骤：

1. 进入助记词展示页或绑定 SN 页。
2. 直接刷新 WebView、重启 App，或从 hash 中间路由重新进入。
3. 检查未完成流程状态。

通过标准：

- 未完成的密码、Active Code、用户名检查结果不持久化。
- 不出现半创建 DID。
- 截图 `02-create-abandoned.png`。

### 2.4 SN 用户名本地校验分支

步骤：

在绑定 SN 页分别输入以下用户名，检查提交按钮和提示：

| 输入 | 预期 |
| --- | --- |
| 空字符串 | 不可提交 |
| ` ABCDEFG ` | trim 并转小写后按 `abcdefg` 校验 |
| `1abcdef` | 首字符非法，不可提交 |
| `abc` | 长度不足，不可提交 |
| `abcdef-` | 尾字符非法，不可提交 |
| `abc.def` | 点号非法，不可提交 |
| `abc_defg` | 下划线非法，不可提交 |
| `abc-def1` | 合法，进入远端检查 |

通过标准：

- 本地非法输入不会触发提交。
- 合法输入会触发 500ms debounce 后的 `auth.check_username`。
- 截图 `02-username-validation.png`。

### 2.5 密码和 Active Code 本地校验分支

步骤：

分别覆盖：

- 密码为空。
- 密码长度小于 6。
- 两次密码不一致。
- Active Code 为空。
- Active Code 非空但远端失败。

通过标准：

- 密码非法时不提交。
- Active Code 为空时不提交。
- Active Code 远端失败时按钮不可用或提交被拦截，并允许修改。
- 截图 `02-password-active-code-validation.png`。

### 2.6 SN 远端失败和重试分支

步骤：

通过 mock SN 依次覆盖：

1. `auth.check_username` 返回不可用。
2. 修改用户名后返回可用。
3. `auth.check_active_code` 返回无效。
4. 修改 Active Code 后返回有效。
5. `auth.register` 返回失败。
6. `user.bind_owner_key` 返回失败。
7. `device.get_by_pk` 前 N 次无 `user_name`，后续成功。
8. `device.get_by_pk` 一直无 `user_name` 直到 20 次超时。

通过标准：

- 前置检查失败时仍停留当前页且可重试。
- 注册失败、绑定失败、轮询超时均不创建本地 DID。
- 轮询最终成功时创建本地 DID 并写入 SN 状态缓存。
- 失败分支均有可读错误反馈。
- 截图 `02-sn-check-failed.png`、`02-sn-register-failed.png`、`02-sn-poll-timeout.png`。

## 3. 导入 DID 流程

自动化等级：可完全自动化，前提是使用 mock SN；真实 SN 冒烟为可部分自动化。

### 3.1 已绑定 SN 助记词导入成功

步骤：

1. 使用 2.1 创建成功的助记词和用户名。
2. 清理本地 App 数据，但保留 mock SN 中按公钥可查到的 `user_name`。
3. 从欢迎页进入导入账户页。
4. 输入助记词、密码、确认密码。
5. 提交。
6. 断言调用 `derive_bucky_public_key`、`device.get_by_pk`、`import_did`。
7. 断言进入 `/main/home`。

通过标准：

- 导入页不要求手动输入昵称或 SN 用户名。
- 本地昵称使用 SN 返回的 `user_name`。
- 导入成功后自动成为 Active DID。
- SN 状态缓存被刷新。
- 截图 `03-import-success.png`。

### 3.2 助记词输入实时校验

步骤：

1. 输入非法单词。
2. 输入合法前缀但最后一个单词仍在输入中。
3. 输入以空格结尾的完整单词列表。
4. 提交校验 checksum 错误的词组。

通过标准：

- 已完成单词会被校验并提示非法单词。
- 未完成的最后一个单词不被过早判定。
- 最终提交以 Rust BIP-39 parse 结果为准。
- 截图 `03-import-mnemonic-validation.png`。

### 3.3 导入失败分支

步骤：

分别覆盖：

- 助记词为空。
- 密码长度小于 6。
- 两次密码不一致。
- SN `device.get_by_pk` 返回无 `user_name`。
- 本地已存在同一 DID，重复导入。
- `import_did` Rust 层返回错误。

通过标准：

- 每个失败分支停留导入页或给出明确错误。
- 无 SN 记录时不得创建本地 DID。
- 重复导入被拦截。
- 截图 `03-import-no-sn-record.png`、`03-import-duplicate.png`。

## 4. 主应用、OOD 和 App 列表

### 4.1 OOD 未绑定状态

自动化等级：可完全自动化。

步骤：

1. 使用没有 `zone_config` 的 Active DID 启动。
2. 进入 `/main/home` 和 `/main/home/ood-activate`。
3. 检查标题、说明、插图、扫描按钮、远程激活禁用按钮。
4. 检查 TabBar 显示规则。

通过标准：

- 未绑定时展示 OOD 激活入口。
- 远程激活按钮为禁用态。
- `/main/home/ood-activate` 显示 TabBar。
- 截图 `04-ood-unbound.png`。

### 4.2 OOD 已绑定状态和解绑

自动化等级：可完全自动化，前提是 mock SN 支持 `zone.unbind_config`。

步骤：

1. 准备 Active DID 的 `sn_status.zone_config`。
2. 进入首页。
3. 断言展示已绑定状态、进入应用按钮和解绑入口。
4. 点击解绑。
5. 在确认弹窗中取消一次。
6. 再次点击解绑并确认。
7. mock SN 断言收到带签名 token 的 `zone.unbind_config`。
8. 断言本地 SN 状态缓存清除或 `zone_config` 被清空。

通过标准：

- 取消解绑不改变状态。
- 确认解绑成功后 UI 回到未绑定入口。
- 解绑失败时状态不被误清。
- 截图 `04-ood-bound.png`、`04-ood-unbind-success.png`。

### 4.3 OOD 扫描设备

自动化等级：可部分自动化。

不可完全自动化原因：真实局域网 OOD 设备、网络拓扑和设备响应不可由 App 单独控制。使用 mock HTTP 设备服务时可完全自动化扫描逻辑；真实设备激活只能作为环境联调。

步骤：

1. mock `local_ipv4_list` 或在可控网络内准备地址。
2. 准备一个返回有效 `active_url` 的 mock OOD：`http://{ip}:3182/device`。
3. 准备一个无效设备、超时设备、相对路径 `active_url` 设备。
4. 进入扫描页。
5. 启动扫描。
6. 检查只展示有效设备。
7. 点击完整 URL 设备。
8. 点击相对路径设备，检查拼接为 `http://{display_ip或ip}:3182/{path}`。

通过标准：

- `127.*` 被过滤。
- 存在非 `172.*` 地址时优先排除 `172.*`。
- 只展示包含有效 `active_url` 的设备。
- 移动端点击设备进入 `/web-container?embedded=1`。
- 桌面端点击设备打开或聚焦独立 WebView 窗口。
- 截图 `04-ood-scan-list.png`、`04-ood-device-opened.png`。

## 6. Runtime、WebView 和 BuckyApi

### 6.1 嵌入式 WebView 测试页

自动化等级：可完全自动化。

步骤：

1. 进入 `/main/setting/embedded-webview`。
2. 检查 iframe 加载 `/test_api.html`。
3. 在 iframe 内调用 `getPublicKey`。
4. 调用 `getCurrentUser`。
5. 调用 `signJsonWithActiveDid`，输入正确密码。
6. 调用 `openExternalUrl`。

通过标准：

- iframe 内存在 `window.BuckyApi`。
- 所有接口返回统一结构 `{ code, message?, data? }`。
- `getPublicKey` 返回 Active DID 第一枚 Bucky wallet 的 public key。
- `getCurrentUser` 返回 DID、public key、username、`sn_username`。
- 签名成功返回 signatures 数组；能读取 SN 用户名时返回本次密码对应的 `pwd_hash`。
- `openExternalUrl` 在当前平台按 WebView 规则打开。
- 截图 `06-embedded-webview-api.png`。

### 6.2 BuckyApi 错误分支

自动化等级：可完全自动化。

步骤：

分别覆盖：

- 无 Active DID 时调用 `getPublicKey`。
- 无 Active DID 时调用 `getCurrentUser`。
- `signJsonWithActiveDid([])`。
- `signJsonWithActiveDid` payload 中包含非对象条目。
- 签名密码错误。
- 用户取消签名弹窗。
- 同时发起两个签名请求。

通过标准：

- 空输入返回 `NoMessage`。
- 非对象条目被过滤。
- 密码错误返回 `InvalidPassword`。
- 取消返回 `Cancelled`。
- 并发请求返回 busy 类结果或只有一个流程被允许继续。
- 错误结果不泄露助记词、密码和私钥。
- 截图 `06-buckyapi-error-branches.png`。

## 8. SN 接口兼容性

自动化等级：可完全自动化，使用 mock SN 服务。

步骤：

1. 将 App `config.json` 的 `sn_host` 指向 mock SN base URL。
2. 检查路由拼接：
   - root：base URL。
   - auth：`${base}/auth`，若 base 已以 `/auth` 结尾则不重复追加。
   - bns：`${base}/bns`，若 base 已以 `/bns` 结尾则不重复追加。
3. 创建流程中记录所有 JSON-RPC 请求。
4. 导入流程中记录 `device.get_by_pk` 请求。
5. 解绑流程中记录 `zone.unbind_config` 请求。
6. 模拟请求超时。

通过标准：

- `auth.check_username` 参数为 `{ name }`。
- `auth.check_active_code` 参数为 `{ active_code }`。
- `auth.register` 参数为 `{ name, pwd_hash, active_code }`，成功必须包含 `access_token`。
- `user.bind_owner_key` 参数为 `{ public_key }`，token 使用注册返回值。
- `device.get_by_pk` 参数为 `{ public_key }`，以非空 `user_name` 为成功判定。
- `zone.unbind_config` 参数为 `{ user_name }`，token 使用当前 DID 签名 token。
- 每次 SN 请求超时时间为 10000ms。
- 创建轮询最多 20 次，每次间隔约 2000ms。
- 日志 `08-sn-rpc-log.json` 脱敏保存。

## 9. 数据持久化和安全审查

自动化等级：可完全自动化。

### 9.1 Vault 结构和敏感信息

步骤：

1. 创建 DID。
2. 导入 DID。
3. 切换 DID。
4. 备份 DID。
5. 删除 DID。
6. 读取 `wallet.store` 中 `vault` 脱敏快照。
7. 搜索明文助记词、明文密码、私钥 PEM、Active Code、SN token、签名弹窗密码。

通过标准：

- `vault` 包含 `version`、`active_did`、`dids`。
- DID id 格式为 `did:bk:1:<ULID>`。
- seed 只保存 KDF 参数、salt、nonce、cipher。
- 不持久化明文助记词、明文密码、私钥 PEM、SN token、Active Code。
- SN 状态按 DID 维度保存，不串用。
- 删除 DID 后其 SN 状态不残留到其他 DID。

### 9.3 静态敏感日志扫描


通过标准：

# BuckyOS 第三方签名接入需求

## 摘要

当前 BuckyOS App 已经具备核心签名能力。第三方页面如果运行在 BuckyOS App 的 WebView / iframe 中，可以通过宿主注入的 `window.BuckyApi` 请求当前 active DID 完成签名。这个模式已经落地，主要限制是第三方页面必须运行在 App 内，普通外部浏览器网页不能直接调用该能力。

为了支持更多第三方接入场景，后续可以扩展两种模式：

- 同设备浏览器拉起 App 签名：网页通过 deep link 拉起当前设备上已安装的 BuckyOS App，由 App 拉取签名请求、展示确认页并完成签名。该方案适合手机浏览器拉起手机 App，也适合 PC 浏览器拉起 PC 桌面 App，但需要补齐 Tauri deep link、移动端 URL Scheme / App Link、桌面端 single-instance 和结果回传机制。
- PC 网页 + 手机扫码签名：PC 网页创建短时签名会话并展示二维码，手机 BuckyOS App 扫码后拉取待签请求，在手机上确认并签名，再把结果提交回 relay 服务，PC 网页通过 WebSocket、SSE 或轮询收到结果。该方案更适合跨设备场景，不要求 PC 安装 BuckyOS App，建议优先实现。

两类新增模式都不应在 URL 或二维码中携带完整待签 payload，更不能携带密码、助记词或私钥。推荐只传递一次性 `session_id`、请求地址、请求摘要和过期时间。签名完成后，结果通过短时、一次性的 relay session 返回给第三方网页，第三方网页必须自行验证签名，而不是只信任 relay 服务。

综合当前框架能力，推荐路线是：先加固现有 App 内签名的来源校验和确认 UI，再优先实现 PC 网页 + 手机扫码签名，最后实现同设备 deep link 签名。

## 1. 背景

第三方应用需要使用用户在 BuckyOS App 中管理的 DID 私钥完成签名。当前已经实现一种接入模式：第三方页面运行在 BuckyOS App 的 WebView / iframe 中，由宿主 App 注入 `window.BuckyApi`，页面通过宿主桥接能力请求签名。

为了覆盖更多第三方使用场景，还需要评估并规划两种外部签名模式：

1. 浏览器网页申请签名，并尝试拉起当前设备上已安装的 BuckyOS App 完成签名。
2. PC 浏览器网页申请签名，用户使用手机上的 BuckyOS App 扫描二维码并在手机上完成签名。

本文档描述需求、当前状态、可行实现方案和主要风险。

## 2. 目标

- 允许第三方网页在不直接接触用户私钥的情况下请求 BuckyOS App 签名。
- 用户必须在 BuckyOS App 内明确确认签名内容，并完成密码验证或后续认证。
- 第三方网页必须能收到签名结果，并可自行验证签名。
- 方案需要兼容当前技术栈：React 前端、Tauri 2、Rust command、Android / iOS / Windows / macOS 桌面端。
- 方案需要区分同设备签名和跨设备签名，不强行用一种机制覆盖所有场景。

## 3. 范围外事项

- 不允许第三方网页直接读取私钥、助记词或本地钱包数据。
- 不要求浏览器静默检测 BuckyOS App 是否已安装。
- 不要求 PC 网页直接连接手机，也不要求手机和 PC 在同一局域网。
- 不在第一阶段实现完整权限授权体系，只实现一次性签名请求。
- 不把完整待签 payload 放入 URL 或二维码中，避免长度、泄露和篡改问题。

## 4. 当前状态

### 4.1 已实现能力

当前已有 App 内 WebView 签名模式：

- `src/bridges/iframeBridge.ts`
  - 监听第三方 iframe 的 `postMessage`。
  - 支持 `getPublicKey`、`getCurrentUser`、`openExternalUrl`、`signJsonWithActiveDid`。
  - 签名时弹出密码输入框。
  - 调用本地 `signJsonWithActiveDid` 后把结果回传给 iframe。
- `src/features/did/api.ts`
  - 通过 Tauri `invoke("sign_json_with_active_did")` 调用原生命令。
- `src-tauri/src/lib.rs`
  - 已注册 `did::sign_json_with_active_did` 命令。
- `src-tauri/src/did/commands.rs`
  - 已实现基于当前 active DID 的 JSON 签名能力。

这说明核心签名能力已经存在，新增模式主要需要解决：

- 外部请求如何进入 App。
- App 如何展示并确认外部签名请求。
- 签名结果如何安全返回第三方网页。

### 4.2 尚未实现能力

当前项目尚未接入：

- Tauri deep link 插件。
- Android intent-filter / iOS URL Scheme / Universal Link / App Link 配置。
- 桌面端 single instance deep link 接收机制。
- 手机二维码扫描能力。
- PC 网页与手机 App 之间的签名 session relay 服务。
- 签名请求的服务端状态机和一次性结果通道。

当前 Android Manifest 只有 launcher intent-filter，不能被 `buckyos://...` 或 HTTPS App Link 拉起。

### 4.3 当前框架相关判断

当前项目使用 Tauri 2。Tauri 2 官方 deep-link 插件支持 deep link 场景，并提供 `getCurrent()` / `onOpenUrl()` 读取启动或运行时收到的 URL。官方文档同时说明：Windows 和 Linux 桌面 deep link 会作为命令行参数传给新进程；如果希望已有进程接收事件，需要与 single-instance 插件集成。

参考：

- Tauri Deep Linking: https://v2.tauri.app/plugin/deep-linking/
- Deep Link JavaScript API: https://v2.tauri.app/reference/javascript/deep-link/

扫码能力方面，Tauri 官方 barcode-scanner 插件支持移动端二维码扫描能力。若官方插件能力不满足需求，也可以按 Tauri 移动端插件机制用 Android Kotlin / iOS Swift 自定义插件。

参考：

- Tauri Barcode Scanner: https://v2.tauri.app/plugin/barcode-scanner/
- Tauri Mobile Plugin Development: https://v2.tauri.app/develop/plugins/develop-mobile/

## 5. 模式 A：第三方页面运行在 BuckyOS App 内签名

### 5.1 状态

已实现。

### 5.2 适用场景

- 第三方页面由 BuckyOS App 打开。
- 第三方页面运行在 App 的 iframe 或独立 WebView 中。
- 第三方愿意接入 `window.BuckyApi`。

### 5.3 基本流程

1. BuckyOS App 打开第三方页面。
2. 宿主向页面注入 `window.BuckyApi`。
3. 第三方页面调用 `BuckyApi.signJsonWithActiveDid(payloads)`。
4. 宿主展示密码确认弹框。
5. 用户输入密码。
6. Rust command 完成签名。
7. 宿主通过 `postMessage` 把签名结果返回给第三方页面。

### 5.4 主要限制

- 只适用于页面运行在 BuckyOS App 内的场景。
- 普通外部浏览器网页无法直接访问 `window.BuckyApi`。
- 第三方页面和宿主之间需要明确 origin 校验策略，当前桥接实现还需要加强来源限制。

## 6. 模式 B：同设备浏览器拉起 BuckyOS App 签名

### 6.1 需求描述

第三方网页在浏览器中发起签名请求。用户点击“使用 BuckyOS 签名”后，浏览器尝试拉起同一台设备上已安装的 BuckyOS App。App 展示签名请求并完成签名，然后把结果返回给第三方网页。

该模式包含两类设备：

- 手机浏览器拉起同一台手机上的 BuckyOS App。
- PC 浏览器拉起同一台 PC 上的 BuckyOS 桌面 App。

PC 浏览器不能直接拉起用户手机上的 BuckyOS App；跨设备场景应使用模式 C。

### 6.2 可行性

可实现，但当前项目没有现成实现。

当前框架可通过 Tauri deep-link 插件实现 App 被 URL 拉起。移动端需要配置 Android intent-filter、iOS URL Scheme 或 Universal Link / App Link。桌面端需要注册自定义 scheme，并处理 single-instance 场景。

### 6.3 推荐协议形态

不建议把完整待签 payload 放在 deep link URL 中。推荐 deep link 只携带一次性 request id 和请求摘要：

```text
buckyos://sign?request_id=REQ_ID&request_hash=HASH&origin=https%3A%2F%2Fthird.example&callback=https%3A%2F%2Fthird.example%2Fcallback
```

字段说明：

- `request_id`：服务端签名请求会话 ID。
- `request_hash`：待签请求规范化后的摘要，防止 App 拉取到的内容与浏览器展示内容不一致。
- `origin`：发起方域名，用于 App 展示和校验。
- `callback`：签名完成后可打开的回调地址，或仅用于服务端结果查询。

### 6.4 推荐流程

1. 第三方网页创建签名请求。
2. 第三方服务端保存待签 payload，生成 `request_id`、`request_hash`、过期时间。
3. 浏览器打开 `buckyos://sign?...`。
4. BuckyOS App 被拉起。
5. App 解析 deep link。
6. App 根据 `request_id` 从服务端拉取待签内容。
7. App 校验 `request_hash`、过期时间、origin。
8. App 展示签名来源、payload 摘要、风险提示。
9. 用户确认并输入密码。
10. App 调用现有 `sign_json_with_active_did`。
11. App 将签名结果提交到服务端，或打开 `callback` 返回浏览器。
12. 第三方网页通过 callback、WebSocket、SSE 或 polling 获取结果。

### 6.5 需要新增的实现

App 侧：

- 添加 `tauri-plugin-deep-link`。
- 桌面端添加 single-instance 支持，确保已有窗口能收到 deep link。
- Android 添加 intent-filter，支持 custom scheme 或 App Link。
- iOS 添加 URL Scheme 或 Universal Link。
- 前端新增 `/external-sign` 或等价签名确认页。
- 新增 deep link 解析、请求拉取、hash 校验、签名提交逻辑。

第三方 / 服务端侧：

- 创建签名 session。
- 暂存待签 payload。
- 提供请求拉取接口。
- 提供签名结果提交接口。
- 提供 PC 页面等待结果接口。

### 6.6 可能问题

- 浏览器无法可靠静默判断 App 是否已安装，只能尝试打开并提供 fallback。
- custom scheme 可能被其他 App 抢占，安全性弱于 App Link / Universal Link。
- Android App Link 和 iOS Universal Link 需要域名验证和线上配置。
- 桌面端 deep link 在 Windows / Linux 上可能启动新进程，需要 single-instance 集成。
- deep link URL 长度有限，不适合携带完整 payload。
- callback URL 可能泄露签名结果，推荐结果通过服务端 session 获取。
- App 被拉起后如果用户没有 DID、未登录、密码错误或取消，需要有明确错误回传。
- 开发环境调试 deep link 比普通 Web 调试复杂，部分平台需要安装后的包才能完整测试。

## 7. 模式 C：PC 网页请求签名，手机 BuckyOS App 扫码签名

### 7.1 需求描述

用户在 PC 浏览器打开第三方网页并发起签名请求。网页展示二维码。用户用手机上的 BuckyOS App 扫描二维码，在手机上确认并签名。PC 网页随后收到签名结果。

### 7.2 可行性

可实现，并且是推荐优先实现的跨设备方案。

原因：

- 不要求 PC 安装 BuckyOS 桌面 App。
- 不依赖浏览器 deep link 行为。
- 不要求手机和 PC 在同一局域网。
- 手机 App 是私钥所在设备，用户确认路径清晰。
- 与钱包类产品的扫码签名心智一致。

### 7.3 结果回传方式

PC 网页不应等待手机直接连接 PC。推荐使用签名 session relay 服务：

1. PC 网页创建签名 session。
2. PC 网页展示二维码。
3. PC 网页通过 WebSocket、SSE 或 polling 等待 session 结果。
4. 手机 App 扫码后从 relay 服务拉取请求。
5. 手机 App 签名后把结果提交回 relay 服务。
6. relay 服务通知 PC 网页。

推荐等待方式：

- WebSocket：优先推荐，实时性好。
- SSE：实现简单，适合单向事件。
- polling：最简单，但延迟和服务端压力更高。

### 7.4 QR 内容建议

二维码不应包含完整 payload。建议只包含 session 信息和摘要：

```json
{
  "type": "buckyos.sign",
  "version": 1,
  "session_id": "REQ_ID",
  "request_url": "https://sign.example.com/session/REQ_ID",
  "request_hash": "sha256:...",
  "origin": "https://third.example",
  "expires_at": 1781512345
}
```

字段说明：

- `type`：二维码类型，避免误扫其他二维码。
- `version`：协议版本。
- `session_id`：签名会话 ID。
- `request_url`：手机 App 拉取请求详情的地址。
- `request_hash`：待签 payload 的规范化摘要。
- `origin`：发起方域名。
- `expires_at`：过期时间。

### 7.5 推荐流程

1. PC 网页调用第三方服务端 `POST /sign-sessions`。
2. 服务端生成 `session_id`、`request_hash`、`expires_at`，保存待签 payload。
3. PC 网页展示二维码。
4. PC 网页订阅 `GET /sign-sessions/{session_id}/events` 或轮询 `GET /sign-sessions/{session_id}`。
5. 手机 BuckyOS App 打开扫码入口。
6. 手机扫码并解析 QR。
7. App 校验 QR 类型、协议版本、过期时间和域名。
8. App 请求 `request_url` 拉取待签 payload。
9. App 规范化 payload 并计算 hash，与 QR 中 `request_hash` 比对。
10. App 展示签名来源、payload 摘要、权限提示。
11. 用户确认并输入密码。
12. App 调用现有 `sign_json_with_active_did`。
13. App 将签名结果提交到 `POST /sign-sessions/{session_id}/result`。
14. 服务端关闭 session，通知 PC 网页。
15. PC 网页收到结果后自行验证签名。

### 7.6 需要新增的实现

App 侧：

- 新增扫码入口和扫码页。
- 接入 Tauri barcode-scanner 插件，或实现自定义 Android / iOS 扫码插件。
- 新增 QR 协议解析与校验。
- 新增外部签名确认页。
- 复用现有签名命令完成签名。
- 新增向 relay 服务提交签名结果的逻辑。

服务端侧：

- 签名 session 创建接口。
- 签名请求拉取接口。
- 签名结果提交接口。
- PC 页面等待结果接口。
- session 过期、取消、已完成、失败状态管理。
- 请求与结果审计日志，但不得记录明文密码、助记词或私钥。

第三方网页侧：

- 创建签名 session。
- 展示二维码。
- 等待并处理签名结果。
- 验证签名结果。
- 处理超时、取消、失败和重试。

### 7.7 可能问题

- relay 服务成为新增基础设施依赖。
- session 必须短时有效，且只能完成一次，避免重放攻击。
- QR 被截图或转发后可能被其他人扫码，需要 App 侧展示来源和请求摘要，并可增加 PC 端确认码。
- 手机签名结果提交前需要校验服务端 TLS 和 `request_hash`。
- PC 网页收到结果后必须自行验证签名，不能只信任 relay 服务。
- 多个手机同时扫码同一个 QR 时，需要 session 状态机只接受第一个有效结果。
- 用户手机没有可用 DID、密码错误、取消签名、网络断开时，PC 网页需要明确展示状态。
- 二维码过期后需要 PC 网页生成新 session，不能复用旧 session。
- 手机 App 需要相机权限，Android / iOS 权限拒绝时需要引导用户开启。

## 8. 安全要求

### 8.1 签名前展示

BuckyOS App 在签名前必须展示：

- 请求来源域名。
- 应用名称或第三方标识。
- 待签 payload 摘要。
- 请求过期时间。
- 使用的 DID / SN 用户名。
- 风险提示。

完整 payload 较长时，可以提供展开查看，但默认展示摘要。

### 8.2 请求校验

App 必须校验：

- session 是否存在。
- session 是否过期。
- `request_hash` 是否匹配。
- payload 是否为可签名 JSON 对象或数组。
- 同一时间是否已有签名请求在进行。

### 8.3 结果校验

第三方网页必须校验：

- 签名是否对应原始 payload。
- public key / DID 是否符合业务预期。
- session id 是否匹配。
- 返回结果是否过期或重复。

### 8.4 敏感信息

以下内容不得进入 URL、二维码、普通日志或第三方页面：

- 明文密码。
- 助记词。
- 私钥。
- 钱包加密材料。

完整 payload 是否可进入服务端日志需要按第三方业务敏感级别单独决定，默认应只记录 hash 和摘要。

## 9. 推荐优先级

### 第一阶段：补齐 App 内签名的安全边界

- 明确 `BuckyApi` 请求来源校验。
- 明确签名前展示内容。
- 统一错误码和取消行为。
- 为后续外部签名复用签名确认 UI。

### 第二阶段：实现 PC 网页 + 手机扫码签名

优先实现模式 C。该方案最符合跨设备场景，且不依赖 PC 安装 App。

最小可用版本：

- PC 生成 QR。
- 手机扫码。
- 手机拉取请求并签名。
- relay 通过 polling 返回结果。

后续增强：

- WebSocket / SSE 实时通知。
- PC 端确认码。
- 更细的权限授权。
- 生物认证。

### 第三阶段：实现同设备 deep link 签名

先支持手机浏览器拉起手机 App，再支持 PC 浏览器拉起 PC 桌面 App。

原因：

- 手机 deep link 使用场景更明确。
- 桌面端需要安装包、协议注册和 single-instance，测试和分发成本更高。

## 10. 验收标准

### 10.1 PC 网页 + 手机扫码签名

- PC 网页能创建签名 session 并展示 QR。
- 手机 BuckyOS App 能扫描 QR。
- 手机 App 能展示请求来源和 payload 摘要。
- 用户确认并输入密码后，手机 App 能成功提交签名结果。
- PC 网页能收到结果并验证签名。
- 过期、取消、密码错误、重复提交、网络失败均有明确状态。
- QR 不包含完整 payload、密码、助记词或私钥。

### 10.2 同设备 deep link 签名

- 手机浏览器能拉起手机 BuckyOS App。
- 桌面浏览器能拉起桌面 BuckyOS App。
- App 冷启动和已运行两种状态都能收到 deep link。
- App 能完成请求拉取、hash 校验、签名确认和结果提交。
- 未安装 App 时，第三方网页能显示 fallback。

## 11. 结论

两种新增模式都具备实现可能。

推荐路线是：

1. 先复用现有签名命令和确认 UI，抽象外部签名请求模型。
2. 优先实现 PC 网页 + 手机扫码签名，通过 relay session 返回结果。
3. 再实现同设备 deep link 签名，移动端优先，桌面端随后补齐 single-instance 和协议注册。

核心签名能力已经存在，主要工程量集中在外部入口、session 服务、请求校验、结果回传和跨平台集成。

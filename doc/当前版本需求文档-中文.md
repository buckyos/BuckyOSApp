# BuckyOS App 当前版本需求文档（中文）

## 1. 文档说明

本文档以当前仓库代码为准，结合 [DID与SN相关流程.md](./DID与SN相关流程.md) 对现有产品边界进行整理。

本文档只描述当前版本已经落地、或已经在代码中明确体现的行为，不补充尚未实现的目标态设计。

适用对象：

- 产品
- 设计
- 开发
- 测试

## 2. 当前版本定位

BuckyOS App 当前承担两类职责：

- 身份宿主：管理本机 DID 的创建、导入、切换、备份、删除，以及与 SN 状态相关的本地缓存。
- Runtime 宿主：为嵌入式 iframe 或独立 WebView 提供受控的宿主能力，并通过 `window.BuckyApi` 暴露有限接口。

当前版本不是完整的应用商店，也不是完整的 OOD 生命周期管理器。它更接近一个“身份入口 + 运行容器”。

## 3. 当前功能范围

### 3.1 已实现范围

- 欢迎页，支持“创建账户”和“导入账户”两个入口。
- DID 说明页。
- 创建流程中的 DID 引导页。
- 助记词生成、展示、确认。
- 创建流程中的 SN 绑定。
- 创建成功页，并继续引导到 OOD 激活入口。
- 导入 DID，并在导入前通过公钥查询 SN 记录。
- DID 列表、身份切换。
- 身份备份。
- 身份删除。
- SN 状态持久化缓存。
- App 列表读取与展示。
- OOD 局域网扫描与激活页打开。
- 嵌入式 WebView 调试页。
- 独立 WebView 打开外部页面。
- `BuckyApi` 注入与三项宿主接口。

### 3.2 明确不在当前范围

- App 安装、升级、卸载。
- 独立的首页 SN 卡片和完整的 SN 状态运营页。
- OOD 激活完成后的本地配置写回编排。
- 更细粒度的权限授权体系。
- 超出 `getPublicKey`、`getCurrentUser`、`signJsonWithActiveDid` 之外的宿主开放接口。

## 4. 核心术语

- DID：本地身份对象，由助记词派生，并包含钱包、公钥、昵称等信息。
- Active DID：当前激活身份。主应用中的身份相关展示与宿主接口都以它为准。
- Bucky Wallet：DID 下的 Bucky 钱包，当前默认使用第一枚钱包公钥作为 SN 查询与签名入口。
- SN：与用户名注册、邀请码校验、公钥查人、zone 信息查询相关的远端服务。
- OOD：用户设备节点。当前版本主要负责扫描设备并打开其激活页。
- Runtime：宿主提供给 Web 页面的运行容器，包含 iframe 方案和独立 WebView 方案。
- `BuckyApi`：注入到页面中的宿主 JS API。

## 5. 产品结构

### 5.1 一级流程

当前应用分为两套主流程：

- DID 流程：`/`、`/create`、`/show-mnemonic`、`/confirm-mnemonic`、`/bind-sn`、`/import`、`/success`
- 主应用流程：`/main/home`、`/main/apps`、`/main/setting` 及其子路由

### 5.2 当前路由

#### DID 流程

- `/`：欢迎页
- `/did-info`：DID 说明页
- `/import`：导入账户
- `/create`：创建流程说明页
- `/show-mnemonic`：展示助记词
- `/confirm-mnemonic`：确认助记词
- `/bind-sn`：绑定 SN
- `/success`：创建成功页

#### 主应用流程

- `/main/home`：首页
- `/main/home/ood-activate`：OOD 激活入口页
- `/main/home/ood-scan`：局域网扫描页
- `/main/apps`：应用列表页
- `/main/setting`：设置页
- `/main/setting/identities`：身份切换页
- `/main/setting/backup`：备份页
- `/main/setting/language`：语言页
- `/main/setting/about`：关于页
- `/main/setting/embedded-webview`：嵌入式 WebView 调试页

### 5.3 TabBar 规则

TabBar 只在以下路径显示：

- `/main`
- `/main/home`
- `/main/home/ood-activate`
- `/main/apps`
- `/main/setting`

其余二级页不显示 TabBar。

## 6. 全局状态规则

### 6.1 启动规则

- 若本地不存在任何 DID，进入 DID 流程。
- 若本地已存在 DID，进入主应用流程。

### 6.2 Active DID 规则

- 创建成功后，新 DID 自动成为 Active DID。
- 导入成功后，新 DID 自动成为 Active DID。
- 如果本地有 DID 但没有 Active DID，主应用会强制跳转到身份切换页。

### 6.3 DID 上下文规则

以下能力都以 Active DID 为准：

- 顶部账户头部显示
- `BuckyApi.getPublicKey`
- `BuckyApi.getCurrentUser`
- `BuckyApi.signJsonWithActiveDid`
- OOD 相关的 SN 状态查询

### 6.4 SN 状态缓存规则

SN 状态按 DID 维度缓存并持久化。

缓存内容当前至少包括：

- `username`
- `zone_config`

前端优先读取缓存；必要时再调用 `device.get_by_pk` 远端查询并刷新缓存。

## 7. DID 与 SN 流程需求

### 7.1 当前创建流程定义

当前代码已经采用“两阶段创建”：

1. 先生成并确认助记词。
2. 再输入 SN 用户名、密码、邀请码并完成 SN 绑定。
3. 只有 SN 绑定成功后，才真正调用 `create_did` 创建本地 DID。

这意味着：

- 当前版本里，“创建 DID”不等于“点击创建按钮后立刻落本地 DID”。
- 本地 DID 的创建发生在 SN 注册与 owner key 绑定成功之后。
- 创建出的 DID 昵称默认使用绑定成功后的 SN 用户名。

### 7.2 欢迎页

页面目标：

- 给用户两个清晰入口：创建账户、导入账户。
- 提供 DID 说明页入口。

页面组成：

- 语言切换
- 主题切换
- App 名称与副标题
- “创建账户”按钮
- “导入账户”按钮
- “DID 是什么？”文本入口

### 7.3 创建流程说明页

页面目标：

- 解释当前流程是“先生成 DID 材料，再绑定 SN”。
- 不要求用户在本页输入昵称或密码。

当前页面行为：

- 展示 DID 说明卡片
- 展示 SN 说明卡片
- DID 卡片可进入本地 DID 说明页
- SN 卡片可打开 `https://sn.buckyos.ai/`
- 点击主按钮后直接生成助记词

当前版本不包含：

- 昵称输入
- 密码输入
- 本地 DID 直接创建提交

### 7.4 助记词展示页

页面目标：

- 展示 12 个助记词。
- 明确提示用户离线备份。

要求：

- 固定顺序展示。
- 明确提示“这是找回账户的唯一方式”。
- 点击“我已完成备份”进入助记词确认页。

### 7.5 助记词确认页

页面目标：

- 确认用户已经真实完成备份。

当前交互：

- 将助记词打乱形成词池。
- 用户按顺序点击单词。
- 顺序完全一致后，允许进入 SN 绑定页。

### 7.6 SN 绑定页

页面目标：

- 完成当前创建流程的第二阶段。
- 同时收集 SN 用户名、本地密码、邀请码。

输入项：

- SN 用户名
- 密码
- 确认密码
- Active Code

当前校验规则：

- 用户名转为小写。
- 用户名长度至少 7。
- 用户名仅允许 `[a-z0-9.-]`。
- 用户名不能包含连续 `..`。
- 密码长度至少 6。
- 两次密码必须一致。
- Active Code 必填。

当前前置检查：

- 用户名输入后延迟调用 `auth.check_username`
- Active Code 输入后延迟调用 `auth.check_active_code`

当前提交顺序：

1. 再次本地校验用户名、密码、邀请码。
2. 调用 `derive_bucky_public_key` 根据助记词派生公钥。
3. 用 `buckyos.hashPassword(username, password)` 生成 `pwd_hash`。
4. 调用 `auth.register`
5. 调用 `user.bind_owner_key`
6. 轮询 `device.get_by_pk`，直到 SN 记录可见或超时。
7. 记录 SN 缓存。
8. 调用 `create_did`，昵称使用用户名，密码使用本页密码。
9. 进入成功页。

当前版本说明：

- 当前并没有独立的“绑定确认页”。
- 当前并没有单独展示 DID Document 绑定页，但文档语义上依赖 `user.bind_owner_key + device.get_by_pk` 的结果确认。

### 7.7 创建成功页

页面目标：

- 告知账户已创建成功。
- 展示 DID 关键信息。
- 引导用户继续进入 OOD 激活入口。

当前展示内容：

- 昵称
- BuckyOS DID
- BTC 地址（如果存在）
- ETH 地址（如果存在）

当前主按钮行为：

- 点击后进入 `/main/home/ood-activate`

### 7.8 导入 DID 页

页面目标：

- 通过助记词恢复一个已经在 SN 上存在记录的身份。

输入项：

- 助记词
- 密码
- 确认密码

当前版本明确不包含：

- 手动输入昵称
- 手动输入用户名

当前校验与导入顺序：

1. 输入过程中调用 `validate_mnemonic_words` 检查已完成单词是否合法。
2. 提交时检查助记词非空、密码长度、两次密码一致。
3. 调用 `derive_bucky_public_key` 派生 Bucky 公钥。
4. 调用 `device.get_by_pk` 查询 SN 记录。
5. 若未查到有效 `user_name`，导入失败。
6. 若查到 `user_name`，使用该名字作为昵称调用 `import_did`。
7. 导入后刷新该 DID 的 SN 状态缓存。
8. 进入 `/main/home`。

结论：

- 当前版本已经实现“导入 DID 不再手动输入名字”。
- 当前版本已经实现“只有 SN 上存在记录的 DID 才允许导入”。

## 8. 主应用页面需求

### 8.1 首页

当前首页实现非常收敛，只承载 OOD 激活入口内容。

当前没有独立的 SN 卡片、SN 注册表单或首页 SN 状态区。

### 8.2 OOD 激活入口页

当前 `/main/home` 与 `/main/home/ood-activate` 都复用同一套 `BindOod` 组件。

页面内容：

- OOD 标题和说明
- 插图
- “扫描本地网络内的设备”按钮
- “激活远程设备”按钮

当前限制：

- “激活远程设备”按钮为禁用态，仅展示“即将支持”。

### 8.3 扫描设备页

页面目标：

- 扫描局域网中可以返回激活页的设备。

当前扫描逻辑：

- 读取本机 IPv4 列表。
- 过滤 `127.*`。
- 如果同时存在非 `172.*` 地址，则优先排除 `172.*`。
- 根据本机网段生成目标地址池。
- 批量调用 `scan_device_batch`。
- 只保留返回了有效 `active_url` 的设备。

当前设备展示：

- 设备名或 IP
- 设备类型
- 基础系统信息（如果存在）
- 显示 IP
- 本机设备标识

点击设备后的行为：

- 若 `active_url` 是完整 `http(s)` 地址，则直接打开。
- 若是相对路径，则拼接为 `http://{display_ip或ip}:3182/{path}` 后打开。
- 通过独立 WebView 打开，不在当前页内跳转。

### 8.4 App 列表页

页面目标：

- 读取并展示当前可用 App 元数据。

数据来源：

- 调用 Tauri 命令 `get_applist`

页面行为：

- 展示图标或首字母占位
- 展示应用名
- 展示描述

当前限制：

- 仅展示，不支持点击启动
- 不支持安装、升级、卸载

### 8.5 设置页

固定入口：

- 语言
- 关于
- 切换身份
- 备份身份
- 删除账户

开发环境入口：

- 打开链接
- 嵌入式 webview

敏感操作规则：

- 备份身份前要求输入密码
- 删除账户前先确认，再输入密码

### 8.6 身份切换页

页面目标：

- 展示全部 DID，并允许切换 Active DID。

当前规则：

- 当前激活身份显示勾选态
- 点击非当前身份后弹出密码框
- 用 `reveal_mnemonic` 作为密码校验
- 校验通过后调用 `set_active_did`

页面还提供“添加身份”入口，底部弹出：

- 创建账户
- 导入账户

### 8.7 身份备份页

进入前置条件：

- 从设置页输入密码并通过 `reveal_mnemonic`

页面流程：

- 展示助记词
- 再次确认助记词
- 完成后返回主流程

### 8.8 删除账户

删除流程：

1. 弹出风险确认框。
2. 再弹出密码输入框。
3. 调用 `delete_wallet` 删除当前 DID。

删除后的跳转规则：

- 若本地仍有 DID，进入身份切换页。
- 若本地已无 DID，返回欢迎页。

## 9. DID 数据需求

每个 DID 当前至少包含：

- `id`
- `nickname`
- `bucky_wallets`
- `btc_addresses`
- `eth_addresses`
- `sn_status`

当前前端默认依赖第一枚 `bucky_wallets[0]` 作为：

- SN 查询公钥
- Runtime `getPublicKey` 返回值
- Runtime 签名默认身份

## 10. SN 模块需求

### 10.1 当前使用的接口

当前前端直接依赖以下 SN 接口：

- `auth.check_username`
- `auth.check_active_code`
- `auth.register`
- `user.bind_owner_key`
- `device.get_by_pk`

### 10.2 当前接口职责

- `auth.check_username`
  用于校验用户名是否可用。
- `auth.check_active_code`
  用于校验邀请码是否有效。
- `auth.register`
  用于提交用户名、`pwd_hash`、邀请码，获取 access token。
- `user.bind_owner_key`
  用于把当前 DID 公钥绑定到用户记录。
- `device.get_by_pk`
  用于按公钥查询 SN 记录，也是导入校验、绑定确认和 zone 状态查询的统一入口。

### 10.3 `pwd_hash` 规则

当前创建流程使用：

- `buckyos.hashPassword(username, password)`

当前 Runtime 签名接口中，如能获取当前 DID 已缓存的 SN 用户名，也会返回同规则计算出的 `pwd_hash`。

## 11. Runtime 与 BuckyApi 需求

### 11.1 Runtime 形态

当前有两种运行形态：

- 嵌入式 iframe
- 独立 WebView 窗口

### 11.2 当前开放接口

- `getPublicKey`
- `getCurrentUser`
- `signJsonWithActiveDid`

### 11.3 接口返回结构

所有接口统一返回：

```ts
{
  code: number;
  message?: string;
  data?: unknown;
}
```

### 11.4 `getCurrentUser`

当前返回字段：

- `did`
- `username`
- `public_key`
- `sn_username`

其中：

- `username` 来自本地 DID 昵称
- `sn_username` 优先来自缓存，缓存不存在时再刷新 SN 状态

### 11.5 `signJsonWithActiveDid`

当前行为：

- 只接受对象数组 `payloads`
- 调用时弹出密码输入框
- 同时只允许一个签名流程进行中
- 成功后返回签名结果数组
- 若能取到 SN 用户名，同时返回 `pwd_hash`

错误场景包括：

- 无 Active DID
- 无可用公钥
- 空消息
- 密码错误
- 用户取消
- 并发冲突

## 13. 当前版本与旧文档的关键差异

以下内容需要作为当前版本共识：

- 创建流程已经是“先助记词，后绑定 SN，再真正创建 DID”。
- 创建页不再输入昵称和密码。
- 绑定 SN 页承担用户名、密码、邀请码三项输入。
- 导入 DID 时不再手动输入名字。
- 导入 DID 必须先查询公钥对应的 SN 记录。
- 首页当前没有独立 SN 模块，主要聚焦 OOD 激活入口。
- `/main/home` 与 `/main/home/ood-activate` 当前实际效果一致。

## 14. 非功能要求

### 14.1 安全

- 助记词本地加密存储。
- 备份、删除、签名、切换身份都需要密码参与或校验。
- 宿主能力不直接暴露到底层私钥。

### 14.2 可用性

- 创建流程应保持短路径。
- 用户名、邀请码检查应有明确状态反馈。
- 网络失败、超时、密码错误需要明确提示。

### 14.3 一致性

- 所有身份相关页面都以 Active DID 为统一上下文。
- SN 缓存与 DID 绑定保存，避免跨身份串用。
- 新增页面或文案时，不应回退到旧版“创建页先输昵称和密码”的逻辑。

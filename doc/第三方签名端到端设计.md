# BuckyOS 第三方签名端到端设计

## 摘要

本文补充第三方签名接入的端到端设计，重点回答三个问题：

1. App 开发者如何通过 SDK 接入 BuckyOS 签名能力。
2. SDK 和 BuckyOS App 内部如何处理运行环境判断、deep link、扫码、relay session、签名确认弹窗和结果回传。
3. 为支持这些流程，BuckyOS App、Web SDK、移动端能力、桌面端能力和 relay 服务分别需要修改哪些地方。

设计上需要明确区分两类请求：

- 授权一个 `session_token`：用于登录、建立短时会话或授权第三方服务在限定范围内访问。弹窗重点展示应用、域名、权限范围和有效期。
- 给特定内容签名：用于对某个明确 payload 做可验证签名。弹窗重点展示内容摘要、签名用途、原始内容查看入口和风险提示。

根据 BuckyOS Security Arch，用户 Global DID / Root Key 不应成为所有日常操作的默认授权方式。因此本方案直接按目标架构设计：`authorizeSession` 由 VerifyHub / Zone 签发短时 JWT；`signContent` 根据业务安全等级选择 Global DID Root Signature 或 Zone Signature。

参考文档：

- [第三方签名接入需求](./第三方签名接入需求.md)
- [BuckyApi 集成指南](./BuckyApi.md)
- [BuckyOS Security Arch](https://github.com/buckyos/buckyos/blob/beta2.2/doc/BuckyOS%20Security%20Arch.md)

## 1. 设计目标

- 第三方 App 不直接接触用户私钥、助记词、明文密码或本地钱包数据。
- 同一套 SDK 能覆盖 App 内 WebView、同设备浏览器拉起 App、PC 网页 + 手机扫码三类路径。
- SDK 对开发者暴露稳定、简洁的 API，内部处理环境判断和 fallback。
- BuckyOS App 对用户展示清晰的确认弹窗，区分“授权会话”和“内容签名”。
- URL、deep link、二维码中不携带完整 payload、密码、助记词或私钥。
- 签名结果返回后，第三方必须自行验证签名或 token 绑定关系，不能只信任 relay。
- 授权会话必须使用 VerifyHub / Zone JWT，内容签名必须明确签名等级，避免把 Global DID Root Signature 用成通用登录态。

## 2. 核心概念

### 2.1 Sign Request

第三方发起的签名或授权请求。统一抽象为：

```ts
type BuckySignRequest =
  | AuthorizeSessionRequest
  | SignContentRequest;
```

### 2.2 授权 session_token

授权 session 的目标不是“给任意业务内容签名”，而是证明用户同意为某个应用建立短时会话。

典型用途：

- 第三方登录。
- 授权当前浏览器会话。
- 允许第三方在限定 scope 内调用后端接口。

推荐结果：

- 由 VerifyHub / Zone 侧签发短时 JWT。
- 第三方验证 JWT issuer、audience、scope、ttl、nonce 和签名。
- 如第三方还需要自己的业务 `session_token`，应在验证 BuckyOS JWT 后由第三方服务端签发，并绑定 BuckyOS JWT 的 subject、audience、scope 和过期时间。

### 2.3 给特定内容签名

内容签名的目标是对明确 payload 产生可验证签名。

典型用途：

- 发布内容。
- 提交交易或订单确认。
- 对应用发布、BNS、DID、收益相关内容做强确认。

推荐结果：

- 关键内容使用 Global DID Root Signature。
- 日常内容使用 Zone Signature。
- 第三方按 DID / Zone DID、公钥、payload hash、session id 和签名等级自行验证。

### 2.4 Relay Session

用于跨进程或跨设备传递请求和结果的短时会话。

Relay 只负责转发和状态管理，不应成为信任根。第三方网页必须验证最终签名。

## 3. SDK 面向开发者的 API

建议由 `buckyos-websdk` 或独立 `@buckyos/sign-sdk` 暴露统一接口。

### 3.1 创建 SDK Client

```ts
import { createBuckySigner } from "@buckyos/sign-sdk";

const signer = createBuckySigner({
  appId: "com.example.notes",
  appName: "Example Notes",
  origin: window.location.origin,
  relayBaseUrl: "https://sign.example.com",
});
```

字段说明：

- `appId`：第三方应用稳定标识。
- `appName`：展示给用户看的应用名称。
- `origin`：当前网页来源。
- `relayBaseUrl`：创建和查询签名 session 的服务地址。

### 3.2 授权 session_token

```ts
const result = await signer.authorizeSession({
  scopes: ["profile:read", "content:write"],
  audience: "https://api.example.com",
  expiresIn: 3600,
  displayMessage: "登录 Example Notes 并授权同步笔记",
});

if (result.ok) {
  const token = result.sessionToken;
}
```

请求语义：

- 用户授权的是一个短时会话，不是对业务正文签名。
- 弹窗应展示应用名称、来源域名、权限范围、有效期和目标服务。
- SDK 返回的 `sessionToken` 是 VerifyHub / Zone 签发的短时 JWT。第三方需要自己的业务 token 时，应在服务端验证该 JWT 后再换发。

### 3.3 给特定内容签名

```ts
const result = await signer.signContent({
  purpose: "publish_note",
  payload: {
    title: "Hello",
    contentHash: "sha256:...",
    createdAt: 1781512345,
  },
  displayMessage: "发布这篇笔记",
});

if (result.ok) {
  const signatures = result.signatures;
}
```

请求语义：

- 用户签的是明确 payload。
- 弹窗应展示内容摘要、payload hash、用途、来源域名和使用的 DID。
- 长内容默认只展示摘要，并提供展开查看入口。

### 3.4 SDK 自动选择通道

SDK 内部按优先级选择通道：

1. 如果当前页面运行在 BuckyOS Runtime，优先使用 `window.BuckyApi`。
2. 如果是移动端浏览器，优先尝试 App Link / deep link 拉起本机 BuckyOS App。
3. 如果是桌面浏览器且检测到开发者选择扫码模式，创建 relay session 并展示 QR。
4. 如果是桌面浏览器且明确启用桌面 App 模式，尝试 deep link 拉起桌面 BuckyOS App。
5. 如果以上均不可用，返回 `Unsupported`，由第三方展示安装或扫码 fallback。

浏览器不能可靠静默判断 App 是否已安装。SDK 只能尝试拉起，并通过超时和页面可见性变化推断 fallback。

## 4. 请求模型

### 4.1 通用字段

```ts
interface BaseExternalRequest {
  protocol: "buckyos.sign";
  version: 1;
  requestId: string;
  requestType: "authorize_session" | "sign_content";
  appId: string;
  appName: string;
  origin: string;
  audience?: string;
  expiresAt: number;
  nonce: string;
  requestHash: string;
}
```

### 4.2 authorize_session 请求

```ts
interface AuthorizeSessionRequest extends BaseExternalRequest {
  requestType: "authorize_session";
  scopes: string[];
  displayMessage?: string;
  challenge: string;
}
```

`challenge` 应由第三方服务端生成，绑定：

- `requestId`
- `origin`
- `audience`
- `scopes`
- `expiresAt`
- `nonce`

BuckyOS App 应将该 challenge 交给 VerifyHub / Zone 认证链路，由 VerifyHub / Zone 签发短时 JWT。第三方服务端验证该 JWT 后，可按自身业务需要换发第三方 `session_token`。

### 4.3 sign_content 请求

```ts
interface SignContentRequest extends BaseExternalRequest {
  requestType: "sign_content";
  signatureLevel: "root" | "zone";
  purpose: string;
  payloads: Record<string, unknown>[];
  payloadSummary: string;
}
```

`signatureLevel` 由第三方声明，并由 BuckyOS App 按策略确认是否允许。应用发布、BNS、DID、收益相关内容必须使用 `root`；评论、点赞、普通消息等日常内容应使用 `zone`。

`requestHash` 应由规范化后的 `requestType + origin + audience + signatureLevel + purpose + payloads + nonce + expiresAt` 计算得到。

## 5. 端到端流程

### 5.1 App 内 WebView 流程

适用场景：第三方页面已经运行在 BuckyOS App 的 iframe 或 WebView 中。

```text
第三方页面
  -> SDK 检测到 window.BuckyApi
  -> SDK 调用 BuckyApi 请求
  -> BuckyOS App 校验来源和请求结构
  -> BuckyOS App 展示确认弹窗
  -> 用户确认并完成认证
  -> BuckyOS App 完成签名或授权
  -> BuckyApi 将结果返回给第三方页面
  -> 第三方页面验证结果
```

需要补齐：

- `window.BuckyApi` 增加结构化外部请求入口，避免只有裸 `signJsonWithActiveDid`。
- iframe bridge 增加 origin allowlist 或至少严格记录和展示来源。
- 统一使用外部签名确认页 / 弹窗。

### 5.2 同设备 deep link 流程

适用场景：浏览器和 BuckyOS App 在同一台手机或同一台 PC 上。

```text
第三方页面
  -> SDK 在 relay / 服务端创建请求
  -> SDK 打开 buckyos://sign?request_id=...&request_hash=...
  -> BuckyOS App 收到 deep link
  -> BuckyOS App 拉取请求详情
  -> BuckyOS App 校验 requestHash、origin、expiresAt、nonce
  -> BuckyOS App 展示确认弹窗
  -> 用户确认并完成认证
  -> BuckyOS App 向 relay / 服务端提交结果
  -> 第三方页面通过 callback、WebSocket、SSE 或轮询收到结果
  -> 第三方页面验证结果
```

deep link 只携带：

- `request_id`
- `request_hash`
- `request_url`
- `origin`
- `expires_at`

deep link 不携带完整 payload。

### 5.3 PC 网页 + 手机扫码流程

适用场景：PC 不安装 BuckyOS App，用户使用手机 BuckyOS App 完成签名。

```text
PC 网页
  -> SDK 创建 relay session
  -> PC 网页展示二维码
  -> PC 网页通过 WebSocket / SSE / 轮询等待结果
  -> 手机 BuckyOS App 扫描二维码
  -> 手机 BuckyOS App 拉取请求详情
  -> 手机 BuckyOS App 校验 requestHash、origin、expiresAt、nonce
  -> 手机 BuckyOS App 展示确认弹窗
  -> 用户确认并完成认证
  -> 手机 BuckyOS App 提交结果
  -> Relay 关闭 session 并通知 PC 网页
  -> PC 网页验证结果
```

QR 内容示例：

```json
{
  "protocol": "buckyos.sign",
  "version": 1,
  "request_id": "REQ_ID",
  "request_type": "sign_content",
  "request_url": "https://sign.example.com/sign-sessions/REQ_ID",
  "request_hash": "sha256:...",
  "origin": "https://third.example",
  "expires_at": 1781512345
}
```

## 6. SDK 内部实现

### 6.1 通道判断

SDK 内部维护通道：

```ts
type SignChannel =
  | "runtime"
  | "mobile_deeplink"
  | "desktop_deeplink"
  | "qr_relay";
```

判断逻辑：

1. `window.BuckyApi` 存在：使用 `runtime`。
2. 移动端浏览器：优先 `mobile_deeplink`，失败后展示 QR 或安装 fallback。
3. 桌面浏览器：默认 `qr_relay`，除非开发者显式启用 `desktop_deeplink`。
4. 用户选择扫码：强制 `qr_relay`。

### 6.2 状态机

SDK 和 relay 统一使用以下状态：

```text
created
  -> waiting_for_app
  -> app_opened
  -> user_confirming
  -> signed
  -> completed
```

失败状态：

```text
expired
cancelled
rejected
invalid_request
invalid_hash
no_active_did
invalid_password
network_error
unsupported
```

### 6.3 SDK 错误码

```ts
type BuckySignErrorCode =
  | "Unsupported"
  | "UserCancelled"
  | "UserRejected"
  | "Expired"
  | "InvalidRequest"
  | "InvalidHash"
  | "NoActiveDid"
  | "InvalidPassword"
  | "Busy"
  | "NetworkError"
  | "NativeError";
```

SDK 应把 App、relay、deep link fallback 的错误统一映射给开发者。

## 7. BuckyOS App 内部实现

### 7.1 外部请求入口

需要新增统一入口：

- Runtime 入口：来自 `window.BuckyApi.requestSignature(...)`。
- Deep link 入口：来自 `tauri-plugin-deep-link`。
- QR 入口：来自扫码页。

三类入口最终都转换为同一个内部模型：

```ts
interface ExternalSignTask {
  source: "runtime" | "deeplink" | "qr";
  request: BuckySignRequest;
  relay?: RelayConfig;
}
```

### 7.2 请求校验

App 必须校验：

- `protocol` 和 `version` 是否支持。
- `requestType` 是否支持。
- `expiresAt` 是否过期。
- `origin` 是否可展示、是否与 request server 绑定。
- `requestHash` 是否匹配。
- `nonce` 是否存在。
- 当前是否已有签名任务在进行。
- 当前是否有 Active DID。

### 7.3 确认弹窗设计

#### authorize_session 弹窗

标题建议：

```text
授权登录
```

必须展示：

- 应用名称。
- 来源域名。
- 授权范围 scopes。
- 有效期。
- 使用的 DID / 用户名。
- 目标服务 audience。

按钮：

- 取消。
- 授权。

风险提示重点：

- 这是授权会话，不是签署具体内容。
- 授权后第三方可在有效期内按 scopes 使用会话。

#### sign_content 弹窗

标题建议：

```text
确认签名
```

必须展示：

- 应用名称。
- 来源域名。
- 签名用途 purpose。
- 内容摘要 payloadSummary。
- payload hash。
- 使用的 DID / 用户名。
- 过期时间。

按钮：

- 取消。
- 签名。

风险提示重点：

- 这是对具体内容签名。
- 签名后第三方可向其他系统证明该内容由当前 DID 确认。

### 7.4 认证和签名

`authorize_session` 和 `sign_content` 必须使用不同的签名链路。

`authorize_session`：

- App 校验请求后，把标准 auth challenge 交给 VerifyHub / Zone 认证链路。
- VerifyHub 完成本地认证后签发短时 JWT。
- JWT 必须包含 `iss`、`sub`、`aud`、`scope`、`nonce`、`iat`、`exp`、`app_id`、`origin`。
- 第三方服务端验证 JWT 后，可以换发自己的业务 `session_token`。

`sign_content`：

- 当 `signatureLevel` 为 `root` 时，App 使用用户 Global DID 私钥完成 Root Signature。该路径必须要求明确的用户确认和本地认证。
- 当 `signatureLevel` 为 `zone` 时，App 使用用户绑定 Zone 的 Zone Signature / VerifyHub 能力完成签名。
- App 必须在弹窗中展示签名等级，避免用户误把日常授权理解为 Root 签名。

`pwd_hash` 不应作为通用第三方签名结果返回。若 SN 兼容流程仍需要，应增加显式 capability 或白名单。

## 8. 授权策略管理

Zone Signature 的便利性来自“首次强确认 + 后续按策略自动或轻量确认”。因此完整方案必须包含授权策略管理，不能只提供一次性签名接口。

### 8.1 首次授权

当第三方首次请求某个 App / origin / scope 组合时，BuckyOS App 必须展示授权确认：

- 应用名称和 `app_id`。
- 来源域名 `origin`。
- 目标服务 `audience`。
- 请求的 scopes。
- 授权有效期。
- 签名等级：Root Signature 或 Zone Signature。
- 后续确认策略：自动、每次弹窗、PIN / 生物认证、必须 Root 确认。

用户确认后，授权策略写入 Zone 的权限事实源。策略存储位置应与 SystemConfig / VerifyHub 的授权模型对齐，不能只保存在第三方网页或 relay 中。

策略示例：

```json
{
  "user_did": "did:bucky:alice",
  "zone_id": "did:zone:alice-zone",
  "app_id": "com.example.social",
  "origin": "https://social.example.com",
  "audience": "https://api.social.example.com",
  "allowed_scopes": ["profile:read", "social:like", "social:comment"],
  "denied_scopes": [],
  "signature_level": "zone",
  "confirmation_policy": {
    "default": "auto",
    "high_risk": "prompt",
    "requires_local_auth": false
  },
  "expires_at": 1781513600,
  "updated_at": 1781510000
}
```

### 8.2 后续自动签名条件

Zone / VerifyHub 只能在同时满足以下条件时自动签名或签发短时 JWT：

- `app_id`、`origin`、`audience` 与授权策略匹配。
- 请求 scope 被允许，且未出现在 `denied_scopes`。
- 请求未过期，nonce 未被使用过。
- payload 类型、purpose 和 `signatureLevel` 符合策略。
- 当前用户仍处于有效登录态，或满足策略要求的本地认证状态。
- 请求未触发风险规则，例如频率异常、scope 升级、目标服务变化、payload 风险升高。

不满足条件时，Zone / VerifyHub 必须拒绝或升级为交互确认。高风险请求必须升级到 BuckyOS App 弹窗，必要时要求 Root Signature。

### 8.3 用户修改或撤销授权

BuckyOS App 或 Zone 管理页必须提供授权管理入口。用户应能按 App、origin、scope、有效期和签名等级管理授权。

至少需要支持：

- 暂停整个应用授权。
- 撤销某个 scope，例如 `social:comment`。
- 缩短或延长授权有效期。
- 将某个 scope 从自动签名改为每次弹窗确认。
- 将某个 scope 从 Zone Signature 升级为 Root Signature。
- 撤销某个 App / origin 的全部授权。

撤销 `social:comment` 后，策略应变为：

```json
{
  "app_id": "com.example.social",
  "origin": "https://social.example.com",
  "allowed_scopes": ["profile:read", "social:like"],
  "denied_scopes": ["social:comment"],
  "signature_level": "zone",
  "updated_at": 1781512345
}
```

之后第三方再次请求 `social:comment` 时：

```text
第三方应用
  -> 请求 social:comment
  -> Zone / VerifyHub 检查授权策略
  -> 发现该 scope 已被拒绝
  -> 返回 permission_denied / scope_revoked
  -> 第三方页面引导用户重新授权，或禁用评论操作
```

### 8.4 撤销的生效边界

授权撤销应立即影响新的签名请求和新的 session token。

已经签出的旧内容不能被“撤回签名”。如果业务需要撤回旧内容，只能通过业务层发布撤回声明、删除内容或标记内容失效。

已经签发但尚未过期的短时 JWT 有两种处理方式：

- 短 TTL：依赖较短有效期自然过期。
- 主动撤销：VerifyHub / Zone 维护 token revocation list 或 session version，服务端验证时检查是否已撤销。

需要强实时撤销的 scope 不应只依赖 JWT 自身过期时间，必须接入撤销检查。

### 8.5 BuckyOS App 的参与边界

BuckyOS App 必须参与：

- 首次授权。
- 授权策略查看、修改、撤销。
- 高风险操作确认。
- 需要 Root Signature 的操作。
- Zone / VerifyHub 要求用户交互认证的场景。

BuckyOS App 不必参与：

- 已授权、低风险、策略允许自动签名的每一次 Zone Signature。
- 已有有效登录态下的短时 JWT 刷新，前提是策略允许且未触发风险规则。

也就是说，BuckyOS App 是用户授权和高风险确认入口；Zone / VerifyHub 是日常策略执行和签发主体。

## 9. Relay 服务设计

### 9.1 API

创建 session：

```http
POST /sign-sessions
```

请求详情：

```http
GET /sign-sessions/{request_id}
```

提交结果：

```http
POST /sign-sessions/{request_id}/result
```

等待事件：

```http
GET /sign-sessions/{request_id}/events
```

也可以提供 WebSocket：

```text
wss://sign.example.com/sign-sessions/{request_id}/ws
```

### 9.2 session 约束

- 默认有效期建议 2 到 5 分钟。
- session 只能完成一次。
- 结果提交后立即关闭。
- 多个 App 同时提交时，只接受第一个有效结果。
- 过期 session 不允许恢复，只能创建新 session。
- relay 日志默认只记录 hash、状态和错误码，不记录明文密码、助记词、私钥或完整敏感 payload。

### 9.3 relay 信任边界

Relay 不是信任根。

第三方网页必须验证：

- `request_id` 是否匹配。
- `request_hash` 是否匹配。
- 签名是否对应原始 payload 或 auth challenge。
- DID / public key 是否符合业务预期。
- `expiresAt`、`nonce` 是否有效。

## 11. 完整实现范围

完整实现应一次性覆盖以下能力：

1. SDK 同时支持 `authorizeSession` 和 `signContent`。
2. SDK 同时支持 Runtime、同设备 deep link、PC 网页 + 手机扫码三类通道。
3. Relay 同时支持 WebSocket、SSE 和 polling，SDK 可按运行环境选择最合适方式。
4. BuckyOS App 支持扫码、deep link、Runtime 三类入口，并统一转换为外部签名任务。
5. BuckyOS App 提供两类确认弹窗：授权会话弹窗和内容签名弹窗。
6. `authorizeSession` 使用 VerifyHub / Zone JWT。
7. `signContent` 支持 Root Signature 和 Zone Signature，并按 `signatureLevel` 展示不同风险提示。
8. 第三方服务端能验证 VerifyHub / Zone JWT、Root Signature 和 Zone Signature。
9. 全链路实现 request hash、nonce、过期时间、一次性 session、并发保护和结果验证。
10. 授权策略管理，支持用户查看、修改、暂停和撤销 App / origin / scope 授权。
11. App Link / Universal Link、custom scheme、桌面 single-instance 和移动端扫码能力全部接入。

## 12. 风险和待决策问题

- VerifyHub / Zone JWT 的 issuer 发现、issuer 公钥获取和轮换机制需要与 SystemConfig 规格对齐。
- `sign_content` 的 `signatureLevel` 策略需要形成明确规则，避免第三方把低风险操作伪装成 Root 签名，或把高风险操作降级为 Zone 签名。
- `pwd_hash` 是否继续对第三方开放。如果继续开放，需要限制 capability 和适用场景。
- AppID 与 origin 的绑定关系需要定义。仅展示 origin 不等于完成 AppID 鉴权。
- 授权撤销是否要求强实时生效。如果要求强实时，需要 token revocation list 或 session version，而不能只依赖短 TTL。
- 自动签名策略必须有风险控制，否则恶意 App 可能在授权范围内高频触发 Zone Signature。
- Relay 是否由 BuckyOS 官方提供，还是第三方自建，需要影响 SDK 默认配置。
- App Link / Universal Link 需要域名资产和线上配置，不能只靠客户端实现。

## 13. 结论

端到端方案应以 SDK 统一入口为核心，对开发者暴露 `authorizeSession` 和 `signContent` 两个语义清晰的 API。

底层通道可以按环境选择 Runtime、deep link 或 QR relay，但进入 BuckyOS App 后必须收敛为同一种外部签名任务模型。

完整方案应同时交付授权会话、内容签名、Runtime、deep link、扫码和 relay 能力。实现顺序可以按工程排期安排，但协议和产品设计不应退化为只支持某一个临时子集。

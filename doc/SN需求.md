新 SN 用户中心产品需求文档
版本：V0.1（基于口述整理）
状态：需求草案
文档目标：统一新 SN 用户中心的产品定位、模块边界、核心流程、接口要求与阶段规划，为后续产品、后端、钱包端与合约设计提供统一依据。

项目背景
过去 SN 的设计逻辑是“按需激活”：
用户只有在明确使用 SN 提供的能力时，才需要输入激活码。
典型场景包括：SN 流量中转、SN DDNS、SN 二级域名等。
在这个阶段，SN 更像一个可选的中心化增强组件。
但随着系统能力增加，越来越多功能在不同环节会依赖 SN，导致用户频繁遇到“点到某个功能时才突然被要求先激活 SN”的情况。该模式已经造成明显问题：
用户无法提前理解为什么自己要先完成 SN 激活。
SN 入口过于分散，认知成本高。
很多功能依赖的并不是“中心化服务本身”，而是更底层的“身份开户能力”。
因此，需要重新定义 SN 用户中心。

产品定位
2.1 核心定位
新 SN 用户中心，本质上是 BNS 合约中 People / Owner 这一类 DID Object 的管理中心。
它解决的不是“所有 SN 功能的集合”，而是：
用户如何在整个去中心化体系中开户；
用户如何注册一个友好名字；
用户如何将名字绑定到 DID Document；
用户如何围绕名字而不是地址进行后续使用、迁移、恢复与管理。
2.2 关键结论
中心化服务是可选的，生态开户不是可选的。
即使用户完全选择去中心化方案，例如：
不使用平台 DDNS；
不使用平台流量转发；
不使用平台域名；
机器本身就是 VPS；
只要要进入整个 BackOS / BNS / SN 体系，仍然需要先完成一次 SN 合约激活 / BNS 开户。

产品目标
3.1 目标
将 SN 从“分散触发的隐藏前置条件”升级为“清晰、前置、可理解的基础能力入口”。
让用户以 名字 进入体系，而不是以长地址进入体系。
保证当前用户中心提供的能力，未来都可以平滑迁移到合约体系。
支持 Web2 到 Web3 的过渡，降低早期使用门槛。
为后续密钥迁移、权限迁移、账户恢复、冻结保护提供统一入口。
3.2 非目标
以下能力不属于本阶段 SN 用户中心的核心职责：
DDNS 配置本身；
流量中转配置本身；
二级域名具体业务配置本身；
高频变化的用户资料编辑；
头像、简介、动态内容等社交型资料管理。
这些能力可以依赖 SN 用户中心，但不应混入其核心职责中。

设计原则
名字优先：对用户呈现的核心身份单位应是名字，不是地址。
合约兼容：V1 即使存在中心化实现，也不能设计出未来无法迁移到合约的能力。
身份与资料分层：链上 / DID 只承载低频、关键、绑定型信息；动态资料应放在用户自己的 OOD。
过渡优先可用：在合约完全上线前，允许中心化托管式引导，但必须支持最终迁移到用户自持。
连续体验优先：激活流程中尽量减少页面跳转与上下文丢失，尤其是浏览器路径。

核心术语
SN 合约激活：用户进入 SN / BNS 生态的开户过程，不再只是“开通某个中心化功能”。
People / Owner：去中心化世界中的第一类 DID 实体，也是用户中心的核心管理对象。
Name：用户在体系中的友好名字，是后续资产、权限、作品、关系的承载入口。
DID Document：名字对应的身份文档，承载低频、关键的绑定信息。
OOD：用户自己的对象空间，用于存放高频变化的用户当前资料。
Default Zone：从 BNS 读取到的默认数据入口，用于进一步获取用户的当前可变信息。
Active Code：当前阶段用于防刷注册的机制；未来同时代表平台为用户提供的早期链上开户补贴。

产品结构与模块拆分
新 SN 用户中心必须拆成独立模块，而不是把所有 SN 能力混在一起。
6.1 模块一：开户与激活
职责：
完成用户进入 SN / BNS 体系的首次开户；
注册名字；
建立名字与 owner 的初始关系；
承接 active code 引导逻辑。
6.2 模块二：名字管理
职责：
申请名字；
检查名字可用性与合法性；
承接未来名字迁移、多人持有等合约能力。
6.3 模块三：DID 管理
职责：
为名字绑定 DID Document；
管理低频、关键、强绑定身份信息；
不承载高频社交资料。
6.4 模块四：登录与身份接入
职责：
支持钱包登录；
支持用户名密码登录；
支持浏览器激活续接；
为未来联合登录预留结构。
6.5 模块五：用户信息读取
职责：
提供 get_user_did；
提供 get_user_info；
严格区分不可变身份信息与当前可变资料。
6.6 模块六：安全与密钥管理
职责：
支持 Web2 到 Web3 的迁移；
支持密钥迁移；
为后续三密钥、挂失、冻结、恢复预留产品入口。

核心业务逻辑
7.1 SN 激活的本质
SN 激活不再只是“开通 SN 服务”，而是：
用户拥有一个地址；
用户在 BNS / SN 合约中注册一个名字；
用户获得最初可用余额或手续费支持；
用户在合约体系里完成正式开户。
在未来合约体系里，平台可通过 active code 完成链上引导：
平台账户支付手续费；
名字归属于用户指定地址；
用户获得进入生态的第一笔基础资源。

7.2 为什么必须围绕名字而不是地址
系统不应鼓励用户长期直接与地址打交道，而应鼓励用户与名字打交道。原因包括：
名字更适合作为用户入口；
更利于密钥更换；
更利于 owner 转移；
更利于局部资产出售与整套身份打包转移。
例如用户出售作品时，可以：
仅转移作品 owner；
或直接转移整个名字的 owner，从而出售整套身份体系。
因此，名字不是“昵称”，而是生态中的核心身份容器。

功能需求
8.1 开户与注册
8.1.1 合约原生注册能力
用户中心需要围绕合约原生 register 能力设计。
核心要求：
输入合法且可用的名字；
合约校验名字合法性与可用性；
支付足够手续费；
注册成功后，默认将当前调用地址设为该名字的 owner。
同时需要支持平台代付场景：
手续费由平台账户支付；
名字 owner 可以直接指定为用户地址；
用于承接 active code 的早期引导逻辑。
8.1.2 临时接口：register_by_password
在合约全面接管前，需要保留过渡接口：
register_by_password(name, password_hash, active_code)
作用：
为不熟悉钱包的用户提供低门槛入口；
保持 Web2 用户的使用习惯；
作为 Web2 → Web3 的过渡桥梁。
本质上，这代表平台暂时帮助用户托管一把密钥或一层身份控制能力。

8.2 登录与过渡
8.2.1 用户名密码体系长期有价值
即使从长期看，用户名密码体系仍有价值，因为它能：
支持普通用户低门槛进入；
支持 Web2 → Web3 迁移；
支持恢复、挂失、冻结等安全能力；
承接一部分普通用户对中心化辅助保护的需求。
8.2.2 不允许用电子邮件替代名字
系统可支持不同登录方式，但用户在体系里必须有一个正式名字。
也就是说：
登录方式可以是密码，未来也可以是联合登录；
但体系内的身份核心必须是名字；
电子邮件不能成为主身份标识。

8.2.3 用户迁移承诺
对于通过用户名密码进入系统的用户，平台必须支持：
在用户准备好之后，
将该名字的所有权，
从平台托管状态，
不可撤回地迁移到用户自己的钱包密钥上。
这是一条基础产品承诺。

8.3 DID 与用户资料边界
8.3.1 DID 只承载低频关键内容
DID Document 中只应放置：
不可变或低频变更的信息；
关键身份绑定信息；
公钥类信息；
默认 zone / binding 信息；
其他不能频繁修改的公开身份入口。
8.3.2 高价值操作提示
更新 DID Document 在未来是链上高价值操作，应在产品上明确体现：
强提示；
二次确认；
不做成“随手可改”的普通资料编辑页。
8.3.3 可变资料放在 OOD
头像、简介、当前资料等高频变化信息，不应放在 DID 或链上。
用户当前资料的读取逻辑应为：
从 BNS 读取该用户的 default zone；
通过 default zone 的固定 URL / 固定入口；
读取该用户真实的当前资料。
因此，用户中心不应直接承担高频资料编辑能力。

8.4 名字规则
V1 需明确固化以下规则：
必须通过统一的合法性校验函数；
名字长度 ≥ 7 个字符；
不允许与已注册名字重复；
存在保留名字 / 黑名单机制；
小于 7 个字符的名字，普通机制下不可注册；
部分高价值名字即使长度满足要求，也应进入保护名单。
高价值名字保护策略：
可基于外部域名价值数据导入黑名单；
例如：长度 ≥ 7 且外部价值超过某阈值的名字；
先保护，后续再设计优先分配给原权益拥有者的过渡机制。

8.5 钱包入口
钱包是平台鼓励的主路径。
8.5.1 钱包创建流程
钱包侧创建账户应拆成两个必须完成的部分：
第一部分：本地创建地址 / 助记词 / 密钥
本地完成，不依赖网络；
一旦完成，用户已拥有地址。
第二部分：完成体系开户
输入唯一名字；
输入用户名密码；
输入 active code；
完成后才算账户真正创建完成。
8.5.2 完成后的状态
账户创建完成后，系统应实现：
该名字与用户钱包密钥绑定；
用户既可以用密码登录；
也可以用密钥登录。
8.5.3 关于“跳过密码”
口述中存在“可跳过但不鼓励”的思路，但当前主路径更明确倾向于：
V1 默认要求完成用户名密码体系；
若未来保留“跳过密码”，则只能作为受限的纯钱包模式，不应成为默认主流程。

8.6 纯浏览器入口
虽然平台整体会强引导用户用钱包，但纯浏览器入口必须提供完整支持，尤其面向传统 NAS 用户。
8.6.1 浏览器激活页的前置步骤
如果激活页在纯浏览器中打开，则不能直接进入激活主流程，而应先进入“账号空间”。
账号空间提供两个动作：
创建新账号；
登录已有账号并继续激活。
8.6.2 为什么必须支持“登录已有账号”
现实中会出现大量激活中断场景：
用户中途离开；
参数没填好；
程序 bug；
测试期失败；
用户走到一半中断。
如果不支持登录已有账号继续激活，会导致：
用户第一步已经注册的账号无法复用；
active code 被误认为失效；
激活体验极差。
因此，浏览器路径必须支持中断恢复。

8.6.3 新接口：login_by_password_and_active_code
建议增加接口：
login_by_password_and_active_code(name, password_hash, active_code)
作用：
在激活页内部完成续接；
避免跳转到外部联合登录页面；
降低用户丢失上下文的风险；
作为当前阶段的轻量安全校验。
8.6.4 统一处理逻辑
激活页不应强行让用户区分“注册”还是“登录”，而应采取统一模式：
用户输入名字 / 用户名、密码、active code；
系统先尝试登录；
若登录成功，则直接进入未完成的激活流程；
若登录失败，但 active code 有效，则转为注册流程；
整体保持在当前激活页内部完成。
这能最大化降低用户困惑。

8.7 三密钥模型（长期规划）
后续合约层建议支持三密钥模式，作为普通用户推荐方案。
8.7.1 三把密钥
用户常用密钥；
用户备用密钥；
平台 / 中立服务密钥。
8.7.2 基本机制
采用 2-of-3 思路：
任意两把密钥可完成重置或恢复一把密钥；
常用密钥用于日常操作；
备用密钥用于恢复；
平台密钥用于辅助恢复、挂失、冻结。
8.7.3 权利边界
平台不能拥有永久控制权；
用户最终可替换掉平台参与的密钥；
平台只是恢复与保护中的辅助角色。
8.7.4 挂失 / 冻结
用户发现主密钥丢失后，可通过用户中心发起挂失：
平台走更严格认证；
触发冻结；
冻结后名字下资产不可继续转移；
待第三把密钥参与后，再重置控制状态。

核心读取接口需求
9.1 get_user_did
读取用户 DID 信息，返回：
名字对应 DID Document；
低频关键绑定内容；
owner / default zone 等关键索引信息。
9.2 get_user_info
读取用户当前信息，但其数据来源原则上不应直接来自 DID 本体，而应：
先通过 BNS 获取 default zone；
再从 default zone 对应入口读取动态资料。

阶段规划
V1
用户中心基本框架上线；
支持名字注册；
支持 register_by_password；
支持钱包入口；
支持纯浏览器入口；
支持中断恢复；
固化名字规则；
支持 DID 基础绑定。
V1.5
优化激活文案；
支持更明确的登录 / 续接状态管理；
预留联合登录结构；
完善 active code 安全策略。
V2
合约注册全面接入；
平台代付 + 用户 owner 指定能力接入；
支持自托管迁移。
V3
三密钥恢复体系；
挂失 / 冻结 / 重置；
更完整的 owner 迁移与名字治理能力。

关键验收点
用户不能只创建地址而不完成名字开户。
钱包路径创建完成后，必须同时支持密钥登录与密码登录。
浏览器路径必须支持中断恢复。
用户重复进入激活页时，系统应自动先尝试登录再尝试注册。
active code 不应因中断流程而被用户感知为“浪费”。
DID 编辑界面必须明确体现其低频、高价值属性。
可变资料不得直接落入 DID 主体。
名字规则必须正确拦截短名、重名、保留名和高价值保护名。
当前中心化实现不能与未来合约模型冲突。

待确认事项
V1 中用户名是否与链上 name 完全使用同一字段。
钱包路径中“跳过密码”是否保留为隐藏能力，还是 V1 直接取消。
高价值名字保护所使用的外部价值数据源与阈值。
浏览器路径是否保留跳外部联合登录作为备用安全方案。
DID Document 的首版字段清单。
三密钥模型在合约层的具体实现方式与冻结语义。

总结
新 SN 用户中心不应再被理解为“某些 SN 服务的激活页”，而应被定义为：
用户进入 BNS / SN / BackOS 去中心化世界的身份开户中心。
它的核心不是“开服务”，而是：
注册名字；
建立 owner；
绑定 DID；
支持从 Web2 平滑过渡到 Web3；
让用户以后始终更多地与名字打交道，而不是与地址打交道。
